CLAUDE.md の「設計判断は ADR に残す」規約に従い、`docs/adr/NNNN-*.md` を作成する。

手順:

1. `docs/adr/` の既存ファイルを確認し、最大の連番 + 1 を 4 桁ゼロ詰めで採番する。
   採番衝突を避けるため、ブランチ間で並行して起票する場合は起票時に既存の最大番号を再確認する。
2. タイトル「$ARGUMENTS」から kebab-case のスラッグを作り、`docs/adr/NNNN-<slug>.md` を作成する。
3. 既存 ADR（例: `docs/adr/0001-tech-stack.md`）のハウススタイルに合わせ、日本語で以下の構成を埋める。
   見出し・メタは下記に統一する（表記揺れを増やさない）:
   - `# ADR-NNNN: <タイトル>`
   - `- ステータス: <制御語彙>`
   - `- 日付: YYYY-MM-DD`
   - `- 関連: [ADR-XXXX](XXXX-<slug>.md)（一言で関係）` … markdown リンクで、関連があれば
   - `## コンテキスト` — なぜこの判断が必要か
   - `## 決定` — 何を採用するか
   - `## 検討したが採用しなかった選択肢` — トレードオフ・却下理由
   - `## 影響` — 観測性・IaC・テストへの波及（CLAUDE.md の原則に沿う）。フォローアップや移行があれば節内に。
4. 作成後、ファイルパスと要点を報告する。設計判断は人間がレビューする前提で、断定しすぎない。

## ステータスの制御語彙

`- ステータス:` は次の 4 値に統一する。実装状況・日付・検証メモは status 行に埋め込まず、本文の `## 影響` へ書く（status 行を機械可読に保ち、索引生成 `just docs-index` を安定させるため）。

- `Proposed` — 提案中（未受理）
- `Accepted` — 受理・有効
- `Superseded by ADR-NNNN` — 後続 ADR に置き換えられた
- `Deprecated` — 廃止（後継なし）

**supersede/改訂は必ず両方向を更新する**: 新 ADR に「ADR-XXXX を置き換える」を書いたら、旧 ADR の status を `Superseded by ADR-<新>` に更新し、`- 関連:` からも新 ADR を指す。片方向で放置すると、索引・status だけ見た読者（および AI エージェント）が失効 ADR を有効と誤読する。
