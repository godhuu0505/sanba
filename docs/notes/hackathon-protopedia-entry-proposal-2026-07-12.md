## 1. 戦略サマリ — どこで勝つか

競合 108 作品中、SRE 自己修復系が最激戦区、技術スタック（Gemini/Cloud Run/ADK）は全員同じ。
SANBA が単独で立てる交差点は次の 3 つで、登録文面はこの 3 点だけを太く通す。

1. **課題**: コードは AI が書ける時代、ボトルネックは要件定義（Findy ドメイン直撃・審査員の実感）
2. **必然性**: 音声 S2S 対話の中で、矛盾検知エージェントが Lead の質問計画を**リアルタイムに書き換える**
   自己検証ループ（単発 LLM では不可能・競合不在）
3. **まわす**: 検知率を評価データセットで CI 回帰し、本番セッションを LLM-judge で採点して
   Cloud Monitoring に流す **LLMOps 実装**（80 作品中ほぼ唯一の深度）

避けること: 「多機能の列挙」（13 レイヤーのスタック自慢は ADR-0008 の反省どおり埋没要因）、
「要件を実装するエージェント」との混同（Provena/FlowForge と同じ棚に置かれない）。

## 2. 提案文面

### タイトル（30字前後・課題と手段が一文で分かる形）

> **SANBA — 会議の前に、AIが要件の"言ってないこと"を音声で引き出す壁打ち産婆**

（現 README の「解像度高く、要件を生み出す音声マルチエージェント」は内向きの言葉。
「壁打ち」「言ってないこと」など審査員が3秒で課題を想起できる語に置き換える）

### 一言サマリ（ProtoPedia の概要欄）

> 生成AIでコードを書く時間は1/10になったのに、要件の抜け漏れ・矛盾は会議室に残ったまま。
> SANBA は PdM が企画会議に持ち込む前に音声で壁打ちする AI 産婆。話すそばから
> 4 つの専門エージェントが「まだ言っていないこと」「さっきと矛盾すること」を検知して
> 問い返し、確定した要件を出所付きドキュメントに落とす。検知率は評価データセットで
> CI 回帰し、本番セッションも LLM-judge で採点して継続改善する——要件定義そのものを
> DevOps のループに載せた。

### ストーリー構成（本文・ピッチ共通の 6 節）

1. **課題**（15秒）: 「AI で実装が速くなるほど、間違った要件が速く形になる」。
   聞き漏らし・暗黙の前提・言った端から矛盾する要望。レビューはコードにはあるのに、
   要件にはない。
2. **解決**（30秒）: 会議の前に SANBA と 1:1 で 10 分話す。人間は話すだけ。
   Lead が問いを立て、非機能・スコープ・矛盾検知の 3 エージェントが並走し、
   「その機能、さっき『ログイン不要』と言った話と矛盾します」と**その場で**割り込む。
   終わると出所メタ付きの要件ドキュメントと GitHub Issue ができている。
3. **エージェントの必然性**（審査観点に一対一対応）: 検知 → Lead の質問再計画 → 深掘り、
   という**エージェント間の相互作用**が核。単発 LLM 呼び出しでは「次に聞くべき問い」を
   会話の履歴と検知結果から自律決定できない。相互作用は Cloud Trace のトレースで
   そのまま見せる（デモのスクリーンショットを掲載）。
4. **つくる**: Gemini Live (S2S) + LiveKit Agents / ADK マルチエージェント
   （root + subagent + agent-as-a-tool）/ Firestore / Elasticsearch RAG（過去セッション
   との矛盾呼び戻し）。音声は I/O 層に隔離し、脳はテキスト——音声不調でもデモが成立する
   設計判断（ADR 68 本で全決定を公開）。
5. **まわす**（最大の差別化・分量を厚く）:
   - 検知率（言葉×言葉の抜け漏れ/矛盾）を評価データセットで定義し、**プロンプト変更の
     たびに Gemini judge が CI で回帰採点、劣化したらマージをブロック**（llm-eval.yml）
   - 本番セッション終了時に LLM-as-a-judge が採点 → 構造化ログ → Cloud Monitoring で
     品質を常時観測
   - OTel 分散トレース（agent/api/worker 全サービス→Cloud Trace）、Four Keys で
     自分たちの開発生産性も計測（dogfooding）
6. **とどける**: <https://youken.sanba.net>（Cloud Run 4 サービス + Global LB +
   Managed SSL、Terraform 一式、WIF 鍵レス CD、非 root 最小コンテナ、gitleaks/Trivy/
   CodeQL/OpenSSF Scorecard）。「審査員がいま触れる」ことを一行目で明言する。

### 効果の見せ方（定量・1 枚）

- 同一シナリオでの Before（人間のヒアリングのみ）/ After（SANBA 同席）の
  **抜け漏れ・矛盾の検知数比較**を 1 枚のグラフに。
- 評価データセットの検知率と、CI 回帰でスコアが守られている様子（llm-eval の
  pass/fail 履歴）をスクリーンショットで。
- 指標は言葉×言葉に限定し、言葉×モック（マルチモーダル）は定性ハイライト 1〜2 例として
  別枠で見せる（ADR-0008 §リスク3。数字を盛らないことを明記すると信頼が増す）。

### システム構成図

README の Mermaid 図をそのまま画像化して掲載。ただしピッチ用には
