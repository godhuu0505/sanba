## 1. イベントの評価軸（確認済み事実と推定）

確認済み: テーマは「**つくる・まわす・とどける**」。審査観点として「アイデアの面白さ」
「エージェントとしての必然性」「自律的に判断しタスクを実行する設計」が公式ブログに明記。
必須技術は Gemini / ADK / Cloud Run 等。審査員による個別フィードバックあり。

推定（過去の Google Cloud 系ハッカソン第1〜3回の構成から）: 課題設定の切実さ・解決策の
有効性が技術力より重く、そこに本回固有の「まわす」（CI/CD・継続改善）「とどける」
（本番品質デプロイ）が加点軸として乗る。過去受賞作（eCOINO・まなびやAI・フクシア等）は
例外なく「**特定職種の切実な業務課題 × 実務フロー適合 × 定量効果**」だった。

## 2. 提出作品の全体傾向

1. **レッドオーシャン: SRE・自己修復系（約2割強）**。CloudMedic / RunGuard / AutoSRE /
   DevOps Lifecycle Agent / AgentOps Platform / DriftScribe…と「Cloud Run 障害を検知→
   診断→修正PR→ポストモーテム」型がもっとも密集。ここは相互に差別化が困難で、
   審査員の目も肥える（＝新奇性で勝ちにくい）。
2. **技術スタックは非差別化要因**。ほぼ全作品が Gemini + Cloud Run（＋ADK）。
   「Gemini を使った」「Cloud Run に載せた」は前提条件であり加点にならない。
   差が付くのは (a) 課題の切実さ、(b) エージェントの必然性の説得力、(c) 「まわす」の実装深度。
3. **「まわす」を語れる作品は少数**。スニペット上で CI/CD・評価ループ・カナリア・
   ロールバックまで踏み込んでいるのは AgentOps Platform、anneal、tech-support、
   スマホおたすけエージェント等ごく一部。**LLMOps（評価データセット CI 回帰・
   オンライン品質スコア）まで実装している形跡がある作品はほぼ見当たらない**。
4. **音声はニッチ**。音声を核にした作品は 6 件程度で、多くは STT/TTS どまり。
   **speech-to-speech（Gemini Live）×マルチエージェントは SANBA 以外に見当たらない**
   （最も近いのは Fugue × biblio: 画面共有×AI専門家会議のデスクトップ音声アシスタント）。

## 3. SANBA の直接競合と相対ポジション

| 競合 | 重なり | SANBA との差 |
|---|---|---|
| Provena (8706) | 「要件を固める」が起点 | Provena は要件確定**後**の自動開発が主役。SANBA は要件を**生み出す過程**（抜け漏れ・矛盾の検知）が主役で、上流で補完関係。ただし「要件」の一言で括られると混同されるリスク |
| FlowForge (8705) | 要件定義書→開発フロー | 同上。既にある文書の解析であり、対話による引き出しではない |
| PhoenixDevOps (8601) | ドキュメント生成 | コードから仕様書を復元する方向。SANBA は人の頭の中から要件を出す方向 |
| FacilitAI (8788) | 会議のリアルタイム支援 | 字幕→コメント投稿の一方向。SANBA は S2S 対話で自ら問いを打つ |
| Fugue × biblio (8764) | 音声×マルチエージェント | デスクトップアシスタント汎用。要件定義という切実な業務課題への特化なし |

**結論: 「音声 S2S × マルチエージェント協調 × 要件の矛盾・抜け検知」という交差点に
競合は不在**。ただし個々の要素（要件・音声・マルチエージェント）は複数作品と被るため、
登録文面で交差点そのものを一撃で伝えないと「要件系の一つ」「音声系の一つ」に埋没する。

## 4. 勝ち筋（審査軸 × 競合分布からの導出)

1. **課題の切実さで先頭集団へ**: 「コード生成が速くなった今、ボトルネックは要件定義」は
   Findy 主催という文脈（開発生産性ドメイン）に正面から刺さる。過去受賞作パターン
   （鮮明な 1 ペルソナの切実な課題）にも一致。ADR-0008 の「社内 PdM の壁打ち産婆」を前面に。
2. **「まわす」で独走**: 競合がほぼ手薄な LLMOps（検知率の評価データセット CI 回帰・
   Gemini judge ゲート・session_scored → Cloud Monitoring）と Four Keys は
   本ハッカソン固有の加点軸に直撃する。**ここは SANBA が確認できた範囲で最も深い**。
3. **エージェントの必然性を「検知→再計画の連鎖」で見せる**: 単発 LLM では不可能な
   自己検証ループ（矛盾検知 Agent が Lead の質問計画を書き換える様子）をトレースで可視化。
   「薄いエージェント」との違いを画面で証明する。
4. **効果の定量化**: 過去受賞作は定量効果の提示が共通項。抜け漏れ/矛盾の検知率
   Before/After を一枚のグラフにする（指標ハックはしない。ADR-0008 §リスク3 の通り
   言葉×言葉に限定）。
5. **審査員が 3 分で体験できる導線**: 公開 URL（youken.sanba.net）は疎通済みだが、
   ゲスト導線とデモ動画が未整備（P0-1/P0-2 残課題）。SRE 系競合はデモが地味になりがち
   （ログとPR）に対し、音声対話は**デモ映えで圧倒できる**数少ない領域。動画の質が合否を分ける。

## 5. リスク

- **音声デモの実演リスク**: ADR-0008 の通り脳はテキストにありフォールバック可能だが、
  ピッチ本番のネットワーク・マイク環境は事前検証必須。
- **「要件系」への埋没**: Provena / FlowForge と同じ棚に置かれると上流/下流の違いが
  伝わらない。「要件を**実装する**エージェントではなく、要件を**産む**エージェント」と
  対比で明示する。
- **未収集 28 件**: 検索に露出していない＝告知していない作品にダークホースがいる可能性。
  7/30 のファイナリスト発表（Google Cloud ブログ掲載・fetch 可能）で答え合わせすること。
