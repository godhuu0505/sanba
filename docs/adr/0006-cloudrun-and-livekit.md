# ADR-0006: Cloud Run 継続（GKE 見送り）と LiveKit 継続（GCPネイティブ代替の見送り）

- ステータス: Accepted
- 日付: 2026-06-14

## コンテキスト
「なぜ GKE でないのか」「なぜ第三者の LiveKit を使うのか」はピッチで必ず問われる。
ハッカソンの勝率という観点（加点とリスクのバランス）で再検討し、決定を記録する。

## 決定 1: 実行基盤は Cloud Run を継続（GKE は採用しない）
**理由（GKE 不採用）**
- 必須要件は Cloud Run / GKE のいずれか 1 つで充足。中井悦司氏は **Cloud Run + Agent Engine を
  「本番志向」として推奨**しており、GKE を選ぶ積極的な加点理由が乏しい。
- GKE が効くのは LiveKit の SFU（メディアサーバ）を自前ホストし、UDP/TCP・常駐・HPA/KEDA を
  自分で握る場合。本構成では SFU を **LiveKit Cloud** に委ねるため、その必要がない。
- 我々の agent worker / api / web は**ステートレス化済み**で、Cloud Run の
  `min/max instances` + `concurrency` で**オートスケール要件を満たせる**。
- GKE は運用負荷・コスト・デモ破綻リスクが大きく、限られた開発期間では割に合わない。

## 決定 2: リアルタイム音声は LiveKit を継続（GCP 単独に代替しない）
**理由（GCP ネイティブ代替の不採用）**
- 多人数音声の **SFU（複数ストリームのルーティング・話者識別）** に相当する**マネージド GCP
  サービスが存在しない**（Google Meet は本用途に組み込めない、自前 WebRTC SFU は高リスク）。
- **1:1 のみ**なら「ブラウザ ↔ WebSocket ↔ Cloud Run ↔ Gemini Live」で LiveKit 無し構成も
  可能。だが本プロダクトの差別化は**多対多**であり、SFU を自作する選択は割に合わない。
- LiveKit は **participant identity / transcription イベント**を提供し、
  「誰の発話か」を要件に紐付ける**多対多のトレーサビリティ**と「適切なログ取得」要件に直結する。
- 必須 GCP 技術（**Gemini / Cloud Run**）は引き続き全面採用するため、要件は充足する。
  SFU 自前ホストを避けるため **LiveKit Cloud** を採用する。

## 検討したが採用しなかった選択肢
- **GKE への全面移行**: 運用コスト・デモ破綻リスク > 加点。却下。
- **GCP のみで WebRTC SFU を自作**: 多人数音声の難所を自前で抱える高リスク。却下。
- **Agent Engine への全面移行**: 有力だが、LiveKit ワーカーの常駐・メディア接続は Cloud Run の
  `min-instances>=1` で足りるため、提出直前に最新の推奨パターンを見て再判断する（保留）。

## 影響
- README / architecture に「Cloud Run 実行・LiveKit Cloud 採用」を明記する。
- ピッチでは「GKE/自作 SFU を避け、本番志向の最小構成で多対多を実現した」と説明できる。
- **MVP（ADR-0008 フォローアップ）**: 1:1 デモでも participant identity による発話の出所トレースを
  1 つ実装し、確定要件に出所メタを残す。1:1 は MVP・多対多は同じ協調パターンの自然な拡張、という
  ストーリーを設計（コード/トレース）で裏付ける。話者識別の本実装は Phase 2/3。

## ピッチ用サマリ（1スライド）
> **なぜ GKE でない?** 必須要件は Cloud Run で充足、ステートレス設計でオートスケールも可能。
> SFU は LiveKit Cloud に委ねるので k8s 運用は不要 —「届けること」に集中するための選択。
> **なぜ LiveKit?** 多人数音声の SFU と話者識別は GCP にマネージドが無い。Gemini Live を
> 多対多に橋渡しし、発話の出所を要件に紐付ける——多対多の必然性そのもの。
