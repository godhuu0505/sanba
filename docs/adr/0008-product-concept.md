# ADR-0008: プロダクトコンセプトの絞り込み（ペルソナ・核・スコープ）

- ステータス: Accepted
- 日付: 2026-06-15

## コンテキスト
SANBA は当初、3 ペルソナ（受託ヒアリング / 社内企画 / 個人開発壁打ち）と 13 レイヤーの
技術スタックを並列に掲げていた。ハッカソンの審査軸は「技術力 < 課題発見・解決策の有効性」で、
過去受賞作は例外なく「困っている人が 1 人に鮮明」。実働 4〜6 週間で全部入りを追うと、
どれも浅くなり「刺さらない技術ショーケース」に着地するのが最大の失敗モード。
本 ADR でディシジョンツリーの根から順に絞り込み、背骨と後景を確定する。

## 決定
コンセプトを次の一文に収束させる。

> **社内 PdM が企画会議に持ち込む*前*に、SANBA と 1:1 で音声対話しながら、要件の抜け漏れと矛盾をリアルタイムに潰す「壁打ち産婆」。**

| # | 論点 | 決定 | 効く理由 |
|---|---|---|---|
| 1 | 主ペルソナ | **社内 PdM / 機能企画** | Findy の開発生産性ドメインに近く審査員の共感を得やすい |
| 2 | 核の一撃 | **リアルタイム矛盾・抜け検知** | grill-me／産婆術の本質。人間の網羅性の限界を超える＝エージェントの必然性 |
| 3 | 音声の位置 | **看板（I/O 層）、脳はテキスト** | speech-to-speech リスクを隔離。音声不調でもテキストでデモ成立（ADR-0002 と整合） |
| 4 | エージェント構成 | **4 エージェント（Lead + 非機能 + スコープ + 矛盾検知）+ Writer tool** | マルチエージェント協調の加点を取りに行く（ADR-0002 のトポロジを踏襲） |
| 5 | 協調の見せ方 | **検知 → Lead 再計画の連鎖をトレース可視化** | 「独立した並列」でなく相互作用が画面で見える |
| 6 | 効果の証明 | **抜け漏れ／矛盾の検知率を指標化**（Before/After + Langfuse 回帰） | 製品価値＝デモ＝「まわす」評価指標を一本化（ADR-0005 と接続） |
| 7 | マルチモーダル | **言葉 × 共有モックの矛盾検知**（核に接続） | 「見る × 聴く」の必然性。"画面では検索がないが今検索したいと言った"（ADR-0004） |
| 8 | デモの場 | **PdM 1 人 × SANBA の壁打ち**（会議前の準備文脈） | 話者識別が不要になり背骨に集中できる |

### スコープ（背骨 / 残す / 後景）
- **背骨（作る）**: 音声 I/O（発話に participant identity の出所メタを付与）→ 4 エージェント協調 →
  矛盾・抜け検知（主指標は言葉同士、言葉 × モックは定性デモ）→ Writer（確定要件に出所メタを保持）→
  **検知率の評価ループ（Langfuse 回帰）** → Cloud Run デプロイ ＋ Cloud Trace/Logging。
- **残す（核に接続）**: マルチモーダル（言葉 × モック検知）、Elasticsearch（過去セッション呼び戻し・根拠付け／スポンサー加点）。
- **後景に降格（記事で言及・拡張扱い）**:
  - Four Keys / DORA ダッシュボード（自分たちの DevOps の dogfooding。製品価値とは斜め）
  - 自前 Grafana / Prometheus / Loki / Tempo（「とどける」の Observability は Cloud Trace/Logging で必要十分）
  - 多人数の話者識別（N:1, N:M。Phase 2/3。同じ協調パターンの自然な拡張として橋渡し）

## 根拠
- ペルソナを 1 人に絞ることで、デモ・差別化ストーリー・実装する機能のすべてが定まる（カスケード）。
- 核を「矛盾・抜け検知」に置くと、エージェントの必然性（自己検証ループ）・デモの効果（"あ、それ言ってなかった"）・
  LLMOps の評価指標（検知率）が**一本の線**で繋がる。
- 音声を I/O 層に隔離し脳をテキスト/マルチモーダルに置くことで、最重量の realtime リスクをデモ全体から切り離せる。
- マルチモーダルを「画像→要件テキスト化」止まりにせず核（検知）に接続することで、独立した便利機能ではなく
  「見る × 聴く」の必然性として効かせる。

## リスクと緩和（フォローアップで決着）
当初は「意識的な保留」として 3 点を残したが、grill-me での詰めにより以下に決着した（本コミットで明文化）。

1. **Elasticsearch を残した重さ** — 核への寄与は「過去セッションとの矛盾呼び戻し」に限定。実装が圧迫されたら
   Elasticsearch の運用スタック（Elastic Cloud / ECK のデプロイ）を諦め、根拠付けは `GroundingStore` に
   実装済みの **in-memory 語重なりフォールバック**で継続する（ADR-0003 のフォールバック）。
   ※当初「Firestore 内検索へ縮退」と書いていたが、ADR-0003 が「Firestore だけで類似検索」を却下しており
   `SessionRepository` に検索 API も無いため、実在しない縮退先だった。実装済みの in-memory 検索に訂正する。
   **縮退トリガ（決定）**: デモ実装の「残り 2 週」マイルストーン時点で、ES による根拠付けが E2E
   （LiveKit ルーム参加〜要件ドキュメント生成）で動作していなければ、その場で ES デプロイを打ち切り
   in-memory フォールバックに固定する。「いつ判断するか」を期限で機械的に固定し、人依存の引き延ばしを防ぐ。
2. **「会議前 1:1」と当初の "多対多" ビジョンの整合** — 記事/ピッチで「1:1 は MVP、多人数会議は同じ協調パターンの
   自然な拡張」と明示的に橋渡しする。
   **設計上の裏付け（決定）**: MVP の時点で発話の出所トレース（LiveKit participant identity）を 1 つ仕込み、
   確定要件にも「誰の発話か」の出所メタを必ず残す。1:1 デモでも内部設計は N:M 前提であることを画面/トレースで示し、
   「結局ただの壁打ちボット？」を設計で否定する。話者識別そのものは Phase 2/3（本表 #8・スコープ節と整合）。
3. **マルチモーダル検知の難度** — 言葉 × モックの矛盾検知はデモ映えするが精度が出にくい。
   **指標の扱い（決定）**: 主指標（ADR-0005 の検知率）は **言葉 × 言葉** の矛盾・抜け検知に限定する。
   マルチモーダル（言葉 × モック）は指標化せず、**定性的なデモハイライト 1〜2 ケース**として別建てで作り込む。
   精度の出ない検知を主指標に混ぜて数字を作為的に底上げすることを避け、原則「指標をハックしない」(CLAUDE.md)を守る。

## 影響
- README のコンセプト節・ペルソナ節を本 ADR の絞り込みに合わせて改訂する（別コミット）。
- ロードマップの Phase 1 を「PdM 1:1 壁打ち × 矛盾・抜け検知 × 検知率評価」に再焦点化する。
- Four Keys / 自前可観測性スタックは「拡張」節へ移す（削除はしない。記事で設計図として言及）。
- 評価データセット（ADR-0005）の主指標を「**言葉 × 言葉**の抜け漏れ／矛盾の検知率」に定義する。
  マルチモーダル（言葉 × モック）検知は主指標から外し、定性デモのハイライトとして扱う。
- 確定要件の各項目に「出所（participant identity）」メタを保持し、UI もしくは Cloud Trace で参照可能にする
  （1:1 MVP でも N:M 設計を示すため）。
