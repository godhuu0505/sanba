# ADR-0037: バックグラウンド先読みと注入ポリシー

- ステータス: Proposed（提案中）
- 日付: 2026-07-05
- 関連: ADR-0002（マルチエージェント・トポロジ）/ ADR-0003（ES grounding）/
  ADR-0008（プロダクト中核＝矛盾・抜け検知）/ ADR-0018（会話体験・会話を止めない）/
  ADR-0024（grill-me 一問一答）/ ADR-0028（repo 紐づけと隔離）/ ADR-0032（end_user モードと出力制御）

## コンテキスト

音声インタビューの体感遅延は、`search_grounding`（embedding 生成 + ES 検索）と
`analyze_requirements`（ADK チームのフル LLM 往復）を **function tool の中で同期実行**している
ことに由来する。ツール結果待ちの間 Gemini Live は沈黙し、web に `deliberating` を出して
沈黙を説明しているのが現状で、産婆術の価値である「問いのテンポ」を毀損している。

一方でコードベースには非同期の足場が既にある: fire-and-forget タスク管理（`_publish` /
`_schedule`）、data channel の非同期 publish（seq / CAS 込み）、検知カードという
**バックグラウンド成果を届ける非音声チャネル**。配線が同期のままなだけである。

## 決定

バックグラウンド実行を次の 3 決定で導入する。

### 決定1: 音声チャネルへの非同期割り込みはしない（恒久ポリシー）

- 採用: **A. 投機的プリフェッチ**（ユーザー確定発話を種に grounding 検索を先読みし、
  ツール呼び出し時に温まったキャッシュを即返す）と **B. 分析のバックグラウンド化**
  （ADK 分析をツール呼び出しから切り離して debounce 実行し、検知カードへ publish する）。
- 不採用: **C. 音声への非同期注入**（検索/分析の完了を契機に `generate_reply` や
  Gemini Live の NON_BLOCKING function calling で会話へ割り込む方式）。
- 会話を中断できるのは**ユーザー起点のバージインのみ**という非対称を保つ。
  バックグラウンド成果の届け先は (1) ツール返答の高速化、(2) web の検知カード、の 2 経路に限る。
  エージェントは自分のターンの流れ（次の `analyze_requirements` / `search_grounding`）で
  最新結果を受け取り、問いに織り込む。

### 決定2: キャッシュは「フィルタ後のみ・メモリのみ・セッション単位」

- プリフェッチは既存の `GroundingStore.search` + 出力制御（ADR-0032 決定8 の allowlist /
  stale・revoked 遮断 / `_allow_repo_grounding` の単一読み）を**通過した後の結果だけ**を保持する。
  生の検索結果はキャッシュに置かない（キャッシュに何が入っていても返してよい、を不変条件にする）。
- キャッシュはプロセス内メモリのみ・セッション単位キー・永続化しない。
- 鮮度: **ユーザー確定発話 2 ターンまたは 60 秒**の短い方で失効。失効・不一致は黙って捨て、
  ツールは同期検索へフォールバックする（**最悪ケース＝現状と同一**が安全性の核）。
- トリガー: partial では発火せず**確定発話のみ**。セッション毎に機内 1 件・latest-wins。
  分析は「確定発話 2 件以上の差分 + 前回から 20 秒以上」の debounce（実質 ≤3 回/分）。
- 失敗は fail-soft（検索 5 秒・分析 30 秒でタイムアウト、リトライなし、ユーザーに不可視）。
  セキュリティ判定のみ fail-closed（既存方針の維持）。

### 決定3: 採否基準を先に固定する（指標をハックしない）

- 必須メトリクス: prefetch のヒット率・短縮時間（同期実測との差分）・staleness 破棄数・
  背景タスク失敗数・分析の実行/スキップ数を構造化ログで計測し、発火元 utterance / session に紐付ける。
- **導入後 2 週間で「ヒット率 ≥ 50% かつツール返答 p95 が 40% 以上短縮」を満たさなければ
  プリフェッチは撤去する**。投機実行のコスト増（embedding / ES / LLM 呼び出し）を
  体感改善で正当化できるかを、機能自体の受け入れ基準とする。

## 理由 / 検討した代替案

- **C（音声割り込み）を採らない理由**: grill-me 流（ADR-0024）は「枝を 1 つずつ、一問一答で
  解消する」規律が本質で、遅延到着した結果による自発的な話題の巻き戻しは問いの主導権を壊す。
  ADR-0018 も「会話を止めずに情報面（タブ/カード）で確認する」体験を設計済みで、
  割り込みはこの思想の逆行。加えて livekit-plugins-google 1.6.0 での NON_BLOCKING 対応は
  未検証という技術リスクもある。再検討の条件は「A+B 適用後もツール待ちの沈黙が
  主要ボトルネックとして計測で残ること」。
- **partial 発火（より早い先読み）を採らない理由**: 日本語 ASR の partial は語形が大きく揺れ、
  embedding 課金を浪費する割にキャッシュキーが安定しない。確定発話でもモデルがツールを
  呼ぶまでに先行時間が取れる。効果が不足すれば debounce 付き partial 発火を追加検討する。
- **キャッシュ読み出し側でのフィルタを採らない理由**: 将来の読み出し経路追加が即漏えい面になる。
  書き込み時点でフィルタ済みなら構造的にすり抜けない（ADR-0032 決定8 と同型の倒し方）。
- **常時バックグラウンド分析（debounce なし）を採らない理由**: ADK 分析は最も高価な LLM 往復で、
  発話毎の実行はコストに見合わない。検知カードの追随は 1 分析周期（p95 < 25 秒目標）で足りる。

## 影響 / フォローアップ

- **観測性**: 上記メトリクスの構造化ログを実装と同時に入れる（CLAUDE.md 原則 3）。
  `deliberating` 表示は同期フォールバック時のみ残す（背景実行は不可視、検知カードの出現が可視化）。
- **IaC**: agent サービスは背景処理が前提になるため、Terraform で CPU always-allocated を
  明示しコメントに理由を残す（暗黙依存にしない）。
- **テスト**: TTL / latest-wins / debounce はクロック注入で決定的にユニットテストする。
  キャッシュヒット経路でも出力制御が不変であることを結合テストで機械検証する。
  Langfuse 評価データセットでの応答品質回帰と Playwright E2E（検知カード表示遅延）は
  フォローアップ issue とする。
- **リリース**: 段階 A → メトリクス確認 → 段階 B の 2 段階。A は失敗モードがキャッシュミスのみで
  検証が容易、B は検知の出方が変わるため分けて観測する。
- **ライフサイクル**: セッション終了時に背景タスクを猶予付きでドレン→キャンセルする
  （現状 `_bg_tasks` は放置されており、本 ADR の実装で解消する）。

## 追補（ADR-0040 §4 との関係・2026-07-06）

本 ADR の決定1「背景分析の結果は (1) 次のツール返答、(2) web の検知カード に限り、**会話への
非同期割り込みはしない**」に対し、ADR-0040 §4 は **アップロード動画の解析完了（`analysis.visual`）に
限って**会話への能動注入を許可する。両者の緊張は次の実装方針で解消した:

- **穏当な注入**: 注入は `session.interrupt()` を伴わない（読み上げ中の割り込みをしない）。
  次の発話境界で自然に織り込ませる（本 ADR が避けた「割り込み」ではなく「機会注入」）。
- **モードゲート**: end_user セッションには注入しない（`claim_video_injection` が
  `allow_repo_grounding` を流用）。grounding 出力制御（ADR-0032 決定8）と同じ境界を保つ。
- **dedup**: 同一素材は 1 回だけ（`_published_gaps` と同じ set パターン `_injected_assets`）。

背景分析（gap/矛盾）の会話注入は引き続き行わない。許可対象は analysis.visual のみに限定する。
