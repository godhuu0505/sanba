# ADR-0030: クロスタブ・ログアウト伝播（BroadcastChannel によるログアウト合図の共有）

- ステータス: Proposed（提案中）
- 日付: 2026-07-04
- 関連: ADR-0012（Google ログイン）/ **ADR-0014 §7（クライアントのセッション持続 — トークンは
  in-memory のみ・localStorage に保存しない）— 本 ADR はこの方針を維持したまま拡張する** /
  ADR-0019（GIS ボタン意匠）/ `apps/web/lib/auth.tsx` / `apps/web/components/RequireAuth.tsx`
- 下敷き: CLAUDE.md「観測できないものは運用できない」「設計判断とレビューは人間が行う」

## コンテキスト

認証要件のひとつに「**ログイン済みユーザーが別ブラウザタブでログアウトした場合、元のタブでも
次のアクションでログインしていないことがわかり、ログイン画面に遷移する**」がある。

現状の構造（ADR-0014 §7）では、Google の ID トークン（credential）は XSS での漏えいを避けるため
**タブごとの in-memory state にのみ**保持し、localStorage 等へ永続化しない。この方針の帰結として、
あるタブでの `signOut()` は他タブに一切伝わらない。他タブは自身の（すでにユーザーが手放したはずの）
credential を持ち続け、authGate（`RequireAuth`）はメモリ上の `loggedIn` しか見ないため、保護
ページを表示し続けてしまう。トークン自体はいずれ期限切れになるが（約 1 時間）、それまでの間
「ログアウトしたのに使えている」状態が残るのは要件に反する。

制約は次の 2 つ:

1. **トークンを共有ストレージに載せない**（ADR-0014 §7 の XSS 方針は不変）。
2. 会話中の API は join 済み `session_token`（契約 §4）で動くため、**idToken の失効と
   「ユーザーの明示ログアウト」は別の事象**として扱う必要がある。

## 決定

**「ログアウトした」という非機微な事実だけを、同一オリジンの他タブへ BroadcastChannel で通知する。**

- チャネル名は `sanba.auth.logout.v1`（`apps/web/lib/auth.tsx` の `LOGOUT_CHANNEL`）。
  メッセージにトークン・PII は載せない（載せる必要のある情報が存在しない）。
- 受信タブは自タブの credential を落とし（`resetLocalAuth`）、`disableAutoSelect()` で One Tap の
  静かな再ログインも止める。credential が落ちると `loggedIn=false` となり、既存の authGate が
  次の描画で `/login?next=` へ送る。**全保護ページが authGate を通っているため、ページ側の
  改修は不要**。
- **伝播するのは明示ログアウトのみ**。`signOut(opts?: { broadcast?: boolean })` は既定で伝播し
  （`/login?loggedOut=1` の明示ログアウト経路）、次の「回復・中断」経路は `{ broadcast: false }`
  で自タブに留める:
  - 401（idToken 期限切れ）からの再認証導線（/admin・/sessions/[id]）
  - サインイン中のキャンセル（/login 画面 12）

  理由: 会話中のタブは `session_token` で動作しており idToken 失効では止まらない。期限切れ回復を
  全タブへ伝播させると、ユーザーが誰もログアウトしていないのに他タブの進行中インタビューを
  authGate 経由で強制終了してしまう。
- dev モード（`GOOGLE_CLIENT_ID` 未設定・AUTH_DEV_BYPASS）は送受信ともスキップする。
- 受信時は `console.info("[auth] cross-tab logout received")` の痕跡を残す（強制ログアウトは
  ユーザーには「突然ログイン画面へ飛ばされた」と見えるため、問い合わせ・デバッグの手がかりを
  最低限確保する。CLAUDE.md 原則3）。
- BroadcastChannel の無い環境（該当するモダンブラウザは実質無い）では**タブ間伝播だけを諦める**。
  自タブのログアウトは常に成立し、他タブもトークン期限切れで最終的に 401 → 再認証に落ちる
  （安全側への劣化）。

## 検討したが採用しなかった選択肢

- **localStorage + `storage` イベント**（初版実装）: 動作はするが、(1) 最終ログアウト時刻が
  localStorage に恒久的に残る、(2) 同値書き込みではイベントが発火しないためユニーク値の細工
  （`Date.now()` + 連番）が要る、(3) プライベートブラウズでの `setItem` 失敗ケアが要る。
  BroadcastChannel はこの 3 点がすべて不要で、2026 年時点で対象ブラウザは全対応 → 置換。
- **トークン自体を localStorage / SharedWorker で共有**: ログイン状態の完全同期（ログイン側の
  伝播も）ができるが、ADR-0014 §7 が退けた XSS リスクを再導入する → 不採用。
- **サーバ主導の失効（トークン失効リスト + push/polling）**: ログアウトの真の失効ができるが、
  失効ストア・接続管理と API 変更が必要で、ハッカソン段階の脅威モデル（トークンは in-memory・
  寿命 ~1h）に対して過剰 → 見送り。将来、セッション管理をサーバ側へ寄せる際に再検討する。
- **401 期限切れ回復も全タブへ伝播**: 「全タブのトークンは同時期に切れるはず」という推定は
  成り立つことが多いが、`session_token` で進行中の会話を巻き添えにする（上記）→ 明示ログアウト
  のみに限定。

## 影響 / フォローアップ

- **ログインの伝播はしない**（要件外）。別タブで再ログインしても他タブは `/login` に留まり、
  各タブでボタン押下（または One Tap）が要る。体験として不足が出たら別 ADR で検討する。
- ログアウトを伝播された保護タブが会話中だった場合、進行中のインタビューは終了する。これは
  「明示ログアウトは全タブで有効」という要件どおりの挙動（意図した強制終了）。
- 観測性: 現状は受信側の `console.info` のみ。素材テレメトリ（`sendTelemetry`）はセッション
  スコープのため流用できない。強制ログアウトの頻度を運用で追いたくなったら、認証イベント用の
  計測経路を別途設計する（#232 の配線と同系）。
- テスト: `apps/web/lib/auth-cross-tab-logout.test.tsx`（伝播・opt-out・受信・close・
  チャネル不在）と `apps/web/lib/auth.test.ts`（dev モードの非購読）で担保。
- クライアントのみの変更で、IaC・API・Terraform への波及は無い。
