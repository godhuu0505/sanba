# ADR-0052: ログイン画面の刷新（NASHI GEN 準拠のクリーン化）と復元中の中立スプラッシュ

- ステータス: Accepted
- 日付: 2026-07-07
- 関連: ADR-0019（**本 ADR がログインボタンの意匠を置換**）、ADR-0025 / ADR-0033（白い紙×原色）、
  ADR-0034（棒人間ロゴ／ファビコン系譜）、ADR-0014 §7（ID トークンを永続化しない）、
  ADR-0012（Google ログイン・信頼境界）、ADR-0030（クロスタブ・ログアウト）、
  ADR-0047（ログイン nonce / トークンリフレッシュ）

## コンテキスト

ログイン画面（`apps/web/app/login/page.tsx`）が、姉妹プロダクト **NASHI GEN** のログイン体験に比べて
以下の点で見劣り・不整合を起こしていた（プロダクトオーナー指摘）:

1. **黒白の入り乱れ。** GIS 純正ボタンを `theme:"filled_black"`（暗色）で描き、周囲を金彩フレームで
   囲んでいた（ADR-0019）。だが ADR-0025 / ADR-0033 で地色は「漆黒×金」→「白い紙（生成り）×原色」に
   刷新済み。生成りの下地に**黒いボタン＋金枠＋白面**が同居し、意匠が濁って見えた。`filled_black` は
   ADR-0019 当時の**暗色基調に合わせた**選択で、現行の明色基調では前提が失効している。
2. **機能しないサインアップ導線。** 左ペインの「今すぐサインアップ」はサインイン枠へスクロールして
   ハイライトするだけの飾りで、実際の登録動作を持たない（登録は Google 初回サインインが兼ねる）。
3. **「ログインし直している」ように見える再訪体験。** SANBA は ID トークンを永続化しない
   （ADR-0014 §7 / XSS 回避）ため、フルロード時は GIS の `auto_select` で credential を取り直す。
   この復元中に保護ページ・`/login` が「ログイン状態を確認しています…」を出していたため、
   ログイン済みの利用者に**毎回ログインさせられている**ように映っていた。NASHI GEN は
   ログイン済みならそのまま画面に入れる。

NASHI GEN のフローは「中央 1 カラム（ロゴ→ワードマーク→タグライン→Google ボタン 1 つ）→ Google →
そのまま画面へ」という最小構成。SANBA もこれに寄せたい、というのが本 ADR のスコープ。

## 決定

### 1. 中央 1 カラムのクリーンなログイン画面（NASHI GEN 準拠）

- `Screen` に上下左右中央寄せの 1 カラムだけを置く: **SANBA マーク（大）→ ワードマーク「SANBA」→
  タグライン「解像度高く、要件を生み出す」→ Google ボタン 1 つ**。
- **廃止:** 2 ペイン（左ブランドペイン）、飾りのサインアップ導線、上部 `AppHeader`、
  クリック後の本人確認スピナー（旧 state 12「Google アカウントを確認しています」＋キャンセル）、
  ログアウト挨拶（旧 state 14「おつかれさまでした」＋「再びログインする」）。
- マークは `BrandMark`（棒人間サンバさん＋山吹の産章 / ADR-0034 系譜）を新設して `Logo`・
  `BrandSplash`・ログイン画面で共有する（実体を 1 つに）。

### 2. Google ボタンは白系の承認バリアントへ戻す（金彩フレーム廃止）

- `apps/web/lib/auth.tsx` の `renderButton` を **`theme:"outline"`（白系）/ `text:"signin_with"`
  （「Google でログイン」）** に変更。白い紙面（ADR-0025）に馴染む Google 承認バリアントをそのまま
  中央に置き、囲みの金彩フレームは廃止する。
- `locale:"ja"` / `size:"large"` / `shape:"pill"`、および GIS script の `?hl=ja` 併用（言語固定）は
  維持（ADR-0019 §フォローアップ 1 の言語固定手順は不変）。
- **ボタン地色・ロゴ・文言は改変しない**（Google ブランド規約・ADR-0012 の信頼境界は不変）。dev の
  「開発用ログイン（bypass）」表記も維持し、純正ボタンと視覚的に区別する。

→ これは ADR-0019 の決定 1（`filled_black`）と決定 2（金彩フレーム）を**置き換える**。ADR-0019 が
`outline` から `filled_black` へ寄せた理由は「暗色基調に馴染ませる」であり、明色基調へ刷新した現行
（ADR-0025 / ADR-0033）ではその理由が失効しているため、`outline` へ戻すのが整合的。

### 3. 復元中は「確認中」ではなく中立スプラッシュ（BrandSplash）を見せる

- 認証解決前（`ready=false` / GIS の静かな再取得中）は、`RequireAuth` も `/login` も
  「ログイン状態を確認しています…」の**ログインし直しを想起させる表示を廃止**し、SANBA マークが淡く
  明滅するだけの中立スプラッシュ（`BrandSplash`、`role="status"` で「読み込み中」）を見せる。復元は
  ほぼ成功する前提なので、**アプリの立ち上がり**としてだけ見せ、そのまま画面に入れたように感じさせる。
- **不変（重要）:** ID トークンを永続化しない方針（ADR-0014 §7）、`auto_select` の静かな再取得、
  ログイン痕跡ヒント（`AUTH_HINT_KEY`）による settle 延長は**一切変えない**。変えたのは**復元中の
  見せ方だけ**で、セキュリティモデル（トークンを localStorage に置かない＝XSS 回避）は不変。
- ログイン済みで `/login` に来たら従来どおり即ホーム（or `?next`）へ `replace` し、ログイン画面を
  見せない。

### 4. ログアウトは挨拶を挟まずクリーンなログイン画面へ

- アカウントメニューは従来どおり `/login?loggedOut=1` へ遷移し、`/login` が実際に `signOut` する
  （ADR-0030 のクロスタブ伝播は不変）。挨拶画面は出さず、そのまま新しいログイン画面を見せる。

### 5. ログアウト意図を cold load でも確実に効かせる（auto_select 抑止）

- 中間画面（旧 state 14）を廃止した副作用として、`/login?loggedOut=1` に**フル/コールドロード**で
  到達したとき（新規タブ・ブックマーク・リロード・モバイルのタブ復元）に、明示ログアウトが無音で
  無効化されるレースが露見した: 新規マウントでは `signOut()` 実行時点で `window.google` が未ロードの
  ため `disableAutoSelect()` が空振りし、その後 GIS の `auto_select` が働くと credential が復元され、
  ログイン画面を見せずにホームへ bounce する（＝ログアウトが取り消される）。旧実装は恒久ガード
  （挨拶画面に留める）でこれを覆っていた。
- 対策（`apps/web/lib/auth.tsx`）: `signOut`/`resetLocalAuth` は **auto_select 抑止の意図をフラグに
  保持**し、GIS が後から利用可能になった初期化（`initializeGis`）でも `auto_select:false` で立ち上げ、
  併せて `disableAutoSelect()` を確実に発火（`g_state` 永続化）させる。明示サインイン（credential
  到着）で抑止は解除し、正常時のリロード復元・能動リフレッシュ（ADR-0047）は壊さない。
- 併せて `/login` は処理後に `?loggedOut=1` を URL から剥がし（History API）、同一タブの後続
  フル再読み込みで同ウィンドウに再入しないようにする（二重の防御）。共有デバイスでのログアウトが
  リロードで戻らないことを保証する（信頼性・セキュリティ）。

## 理由 / 検討した代替案

| 観点 | 採用: 白系 outline + 中立スプラッシュ | 却下案 |
|---|---|---|
| 意匠整合 | 明色基調（ADR-0025）に白系ボタンが馴染む。黒白の濁りが消える | `filled_black` 維持は明色基調で浮く（本 ADR の発端） |
| ブランド規約 | `outline` / `signin_with` は Google 承認バリアント。ロゴ・地色・文言不改変 | フルカスタム金彩化は規約違反（ADR-0019 で既に却下） |
| 再訪体験 | 復元中は中立スプラッシュ＝「ログインし直し」感が消え、NASHI 同等に | トークン永続化で「即表示」も可能だが ADR-0014 §7（XSS 回避）に反するため却下 |
| セキュリティ境界 | 認証ロジック・nonce・信頼境界（ADR-0012 / 0047）は不変。見せ方のみ変更 | — |

- **なぜトークンを永続化しないのか（ADR-0014 §7 を維持）**: ID トークンを localStorage 等に置くと
  XSS で漏えいする。NASHI GEN 相当の「即入場」を**永続化なし**でも十分に近似できる（復元は数百 ms で
  済むのが通常で、その間を中立スプラッシュで覆えば「ログインし直し」には見えない）ため、セキュリティ
  方針は崩さない。

## 影響 / フォローアップ

1. **Web 実装**
   - `apps/web/app/login/page.tsx`: 中央 1 カラムへ全面刷新。旧 state（11/12/14）・2 ペイン・サインアップ・
     ヘッダーを削除。ログイン確立で中間画面を挟まず即 `replace`。
   - `apps/web/lib/auth.tsx`: `renderButton` を `theme:"outline"` / `text:"signin_with"` へ。
   - `apps/web/components/RequireAuth.tsx`: 解決前は `BrandSplash` を表示。
   - `apps/web/components/sanba/BrandMark.tsx`・`BrandSplash.tsx` を新設し `index.ts` から公開。`Logo` は
     `BrandMark` を再利用（実体の一本化）。
2. **テスト**: `login/page.test.tsx`・`login/page.ready.test.tsx`・`RequireAuth.test.tsx`・
   `auth-render-button.test.tsx` を新フロー/新意匠に更新（open-redirect 防止・`?next` 復帰・ログアウト遷移・
   `ready` ゲートの回帰は維持）。
3. **不変**: 認証ロジック（ID トークン検証 / nonce / dev bypass / ADR-0012 信頼境界 / ADR-0030 伝播 /
   ADR-0047 リフレッシュ）は変更しない。
4. **残作業（別スコープ）**: デザインギャラリー（`/design`）の旧ログインモック（「ようこそ戻られました」）は
   実フローと別の陳列であり、本 ADR では触らない。`docs/adr/README.md` 索引の再生成（`just docs-index`）。
