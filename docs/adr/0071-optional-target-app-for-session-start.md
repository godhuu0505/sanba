# ADR-0071: 対象アプリなしでもセッションを開始できるようにする（アプリ選択を任意化）

- ステータス: Accepted
- 日付: 2026-07-12
- 関連: [ADR-0044](0044-home-app-selection-gate.md)（対象アプリの選択を開始ゲートにする — 本 ADR が
  「必須ゲート」を「任意」に改め supersede）/ [ADR-0045](0045-app-slug-urls-and-access-error.md)
  （アプリ従属 URL — slug がある時のスコープ URL は不変、アプリ無しはルート URL 上のクライアント状態で進める）/
  [ADR-0031](0031-product-entity-and-invite-links.md)（product エンティティ・文脈シード）/
  [ADR-0017](0017-figma-canonical-onepath-archaic-copy.md)（一本道の入口フロー）
- きっかけ: オーナー要望（2026-07-12）「対象アプリが無くてもセッションを開始できるようにしたい」。

## コンテキスト

ADR-0044 は「壁打ちは常に “どのアプリの要件か” が起点」という前提で、対象アプリの選択を
01 ホームの開始ゲートにした（未選択なら CTA 無効・候補 0 件なら開始不可で登録導線を案内・
02 準備からは選択 UI を撤去）。この必須ゲートは、登録済みアプリを前提に壁打ちする体験を
なめらかにする一方で、次のケースを塞いでいた。

1. まだアプリを 1 件も登録していないユーザーが、とりあえず壁打ちを始められない
   （0 件なら CTA が永久に塞がる）。
2. 特定のアプリに紐づかない、横断的・探索的なテーマ（「そもそも何を作るべきか」等）で
   要件を引き出したいのに、ダミーのアプリ登録を強いられる。

一方で、必須化は UI の制約に閉じており、バックエンドは既にアプリ任意で成立していた:
`POST /api/sessions` のリクエストは `product_id: str | None = None`、`SessionMeta.product_id`
も任意で、`product_id is None` ならアクセスチェック（`_require_product_access`）を素通しして
アプリ無しのセッションを作る。したがって本要望は **web の入口フローと ADR/テストに閉じた
変更**で満たせる。

## 決定

1. **対象アプリの選択を必須ゲートから任意にする。**
   01 ホームのセレクトに「指定しない」を明示的な選択肢として加え、これを選ぶと
   アプリ無しで「＋ 会話を始める」を活性化する。バッジは「必須」→「任意」に改める。
   - 候補 0 件でもセレクトは無効化しない（「指定しない」が常に選べる）。ADR-0044 の
     「0 件は塞いで登録導線」は取り下げる（登録導線はアプリ管理の常設リンクで足りる）。
   - 候補 1 件の自動選択（ADR-0044）と、slug 未設定アプリを選んだ時の開始不可＋設定導線
     （ADR-0045）は不変。「指定しない」は “アプリを一切紐づけない” 別経路であり、
     slug 未設定アプリの扱いとは区別する。
2. **アプリ無しのセッションはルート URL 上のクライアント状態で進める（新ルートを足さない）。**
   ADR-0045 の `/{slug}/prepare`・`/{slug}/sessions/{id}` は slug がある時のみのスコープ URL
   として維持する。アプリ無しでは準備・開始とも `/` に留まり、step は EntryFlow の
   クライアント状態で持つ（一本道 / ADR-0017 の History API 方式を踏襲）。アプリ無しの
   セッションに紐づく共有可能な対象が無いため、直リンク用の専用ルートは設けない。
3. **02 準備の防衛線（ADR-0044 決定3）はアプリ未確定にのみ効かせる。**
   「指定しない」を選んで準備に入った場合はホームへ巻き戻さない。巻き戻しは
   「アプリを選ぶつもりだったのに未確定（候補 settle 後も未選択・slug 未設定）」の場合に
   限る。判定は「指定しない」センチネルの有無で行う。
4. **アプリを選んだ時の挙動は不変。** 選択済みアプリの文脈シード（用語・説明・
   リポジトリコード）・product_id 送信・slug スコープ URL・アクセスエラー画面
   （ADR-0045 決定3）は従来どおり。

## 検討したが採用しなかった選択肢

- **アプリ無し用に予約 slug（例 `/-/prepare`）を導入する**: 直リンク可能になるが、
  ルーティングと `AccessErrorScreen` ガードの改修で変更範囲が広がる。アプリ無しの
  セッションは紐づく対象が無く直リンクの動機が弱いため、ルート URL 上のクライアント
  状態で足りる。却下。
- **セレクトの空値（「選択してください」）をそのまま “アプリ無し” とみなす**: 追加の
  クリック無しで開始できるが、「まだ選んでいない」と「アプリ無しで進める」を区別できず、
  未選択のまま偶発的に開始する事故を招く。明示的な「指定しない」で opt-in させる。却下。
- **必須のまま “未分類” ダミーアプリを自動作成する**: アプリ一覧を無意味なエントリで
  汚し、文脈シードの前提（実在アプリの用語・リポジトリ）とも噛み合わない。却下。

## 影響

- apps/web `EntryFlow.tsx`: ホームのセレクトに「指定しない」センチネル、CTA ゲートと
  02 の `canStart` を「slug あり or アプリ無し」に緩和、候補 settle 後の巻き戻しを
  アプリ未確定のみに限定、準備画面の対象アプリ表示に「指定しない」を追加。センチネルは
  候補突き合わせ（stale クリア）で消さない。
- 観測性: 巻き戻しの構造化ログ（ADR-0044 影響 / #232）は不変。アプリ無しセッションは
  `product_id=None` として既存のセッション作成ログ・KPI（ADR-0061）に自然に乗る。
- API・agent・DB への変更はない（`product_id` は既に任意）。`createSession` は
  アプリ無しで product_id を送らない。
- テスト: apps/web `app/page.test.tsx` の ADR-0044 由来テスト（0 件で塞ぐ・必須ゲート）を
  本 ADR の挙動へ改訂し、「指定しない」で開始できる経路の回帰を追加する。
- ADR-0045 の slug スコープ URL・存在秘匿・アクセスエラー画面は不変。
