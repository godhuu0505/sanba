# ADR-0005: LLM-as-a-judge オンライン評価と回帰ループ

- ステータス: Accepted
- 日付: 2026-06-14

## コンテキスト
「まわす」軸＝AI を継続的に改善するサイクルを、口先でなく**実装**で示す必要がある。
要件インタビューの品質（非機能要件の網羅・問いの具体性・矛盾検知）を定量化し、
プロンプト/エージェント変更の良し悪しを判定できるようにする。

## 決定
2 系統の評価を持つ。

1. **オンライン評価**: セッション終了時に `score_session` が書き起こしを LLM-as-a-judge
   （Gemini）でルーブリック採点し、Langfuse にスコアを送る。本番の品質を継続観測する。
2. **オフライン回帰**: 代表シナリオのデータセット（`DEFAULT_SCENARIOS`）を CI（`llm-eval`
   ワークフロー）で採点。「良いインタビューが浅いインタビューを上回る」順序関係と
   最低スコア閾値を検証し、劣化したら fail。

Gemini 鍵が無い環境では決定論的なヒューリスティック採点にフォールバックし、パイプラインを
止めない（目的は品質の絶対値でなく**回帰の検出**）。

## 検討したが採用しなかった選択肢
- **人手評価のみ**: スケールせず CI に組み込めない。
- **LLM 判定を本番のみ**: 回帰を事前に止められない → CI のデータセット評価を併設。

## 影響
- `apps/agent/src/kikitori_agent/evaluation.py` を追加、`llm-eval.yml` が回帰を実行。
- ルーブリック/閾値はチューニング対象。スコアは Langfuse Datasets と連携して育てる。
