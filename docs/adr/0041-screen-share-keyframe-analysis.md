# ADR-0041: 画面共有のキーフレーム解析（Live 併用ハイブリッド）

- ステータス: Proposed（提案中）
- 日付: 2026-07-06
- 関連: ADR-0004（マルチモーダル入力）/ ADR-0023（解析進捗の発行設計）/ ADR-0003（Elasticsearch grounding）/
  ADR-0040（アップロード動画の非同期解析）
- 実装計画: [`docs/design/video-and-screen-analysis-implementation-plan.md`](../design/video-and-screen-analysis-implementation-plan.md)

## コンテキスト

画面共有は `RoomInputOptions(video_enabled=True)` で Gemini Live へフレームを転送する配線（ADR-0004）が
あるが、視覚情報の要件化は **LLM が自発的に `note_visual_requirement` ツールを呼ぶことに全面依存**している。

- フレームの決定的な解析・永続化・grounding 投入は存在しない。共有をやめた瞬間、その画面の情報は
  どこにも残らない（LLM がツールを呼んでいた場合のみ断片が残る）。
- native-audio モデルが映像フレームをどの程度解釈しているかは非決定的で、観測もできていない
  （原則3「観測できないものは運用できない」に反する状態）。
- ADR-0004 自身が「映像はコスト増のため共有中のみ有効化する制御を将来入れる」と宿題を明記しており、
  現状は常時 ON のまま。
- カメラと画面共有のトラック選別ロジックも無い。

要件深掘りにおいて画面共有は「この画面のこのボタン」を拾う一次情報であり、**その場の対話反応**と
**セッション後も使える永続記録**の両方が要る。

## 決定

### 1. ハイブリッド構成（Live 併用 + キーフレーム解析）

- **Gemini Live へのフレーム転送は維持**する（即時の対話反応。「いま見えている画面」への応答性を守る）。
  あわせて ADR-0004 の宿題である「**共有中のみ映像入力を有効化**」を実装し、常時 ON をやめる。
- 並走して、agent worker が **`Track.Source.SCREEN_SHARE` の映像トラックを明示的に購読**し、
  キーフレームをバックグラウンド解析して永続記録を作る。カメラトラックは購読対象にしない（選別を明示）。

### 2. キーフレーム抽出はコスト制御を組み込む（変化検知中心）

- フレーム間の**知覚ハッシュ（dHash 等）差分**で「画面が変わったときだけ」を検出する。
- 二重のガード: **最短解析間隔**（既定 10 秒）と**セッションあたり上限枚数**（既定 60 枚）。
  静止画面を見せ続けてもコストが流れず、画面を頻繁に切り替えても上限で頭打ちになる。
- しきい値・間隔・上限は agent の設定値にし、既定 OFF のフラグ（`enable_screen_keyframe_analysis`）で
  段階ロールアウトする。

### 3. 解析と永続化

- キーフレームは既存の vision 経路（`gemini_vision_model`、`analyze_image` と同型のプロンプトを
  画面観察向けに調整: UI 要素・画面名・操作フロー・ドメイン用語・要件候補）で解析する。
- 観察は `index_context(source="screen:{session_id}:{ts}")` で grounding へ投入し、
  `search_grounding` から引けるようにする。あわせて要件候補は既存の視覚要件経路
  （`note_visual_requirement` 相当の永続化、`source_speaker="screen-share"`）に流す。
- **realtime 契約に screen 専用イベントを追加する**: 既存の `analysis.visual` は `asset_id` が必須
  （`apps/web/lib/realtime/parse.ts` の `REQUIRED_FIELDS`、契約 §3）で、素材行への対応付けを前提にしている。
  画面共有の観察には素材（`asset_id`）が存在しないため、`analysis.visual` を流用すると web 側で
  破棄されるか素材行に迷い込む。`analysis.screen(ts, extracted, …)` を契約
  （`docs/design/realtime-contract.md` §3）と `parse.ts` に追加し、web は要件/観察ビューへ反映する
  （疑似 asset_id の捏造はしない — 素材一覧に実体のない行を作らない）。
- **能動注入**: 解析結果はエージェントの会話コンテキストへ「画面から読み取った観察」として注入し、
  深掘り質問（「いま表示されていた◯◯一覧の絞り込み条件は…」）の材料にする。Live が既に言及した
  内容と重複しうるため、注入時に直近の transcript / 記録済み視覚要件との重複を抑制する。

## 理由 / 検討した代替案

- **採用（ハイブリッド）**: 「その場で反応」（Live）と「決定的・永続的な解析記録」（キーフレーム）は
  役割が異なり、どちらか一方では要件深掘りの用途を満たせない。
- **Live 強化のみ（却下）**: プロンプト誘導の改善だけでは非決定的なまま。記録が残らず、観測もできない。
- **キーフレーム解析へ一本化・Live 転送停止（却下）**: コストは下がるが、「画面を見せた瞬間に
  エージェントが反応する」体験を失う。SANBA の音声対話の強みを削る。
- **定期サンプリング（却下）**: 実装は単純だが、静止画面でも課金され続ける。変化検知のほうが
  コストが実利用に比例する。
- **発話連動トリガーのみ（却下）**: 最も安いが、無言での画面遷移（見せながら操作するデモ）を取りこぼす。
  変化検知の補助トリガーとしては将来検討の余地あり。

## 影響 / フォローアップ

- **web / 契約**: `analysis.screen` の追加（`realtime-contract.md` §3、`parse.ts` の許可種別と
  `REQUIRED_FIELDS`、`store.ts` の反映先）。
- **agent**: トラック購読・フレーム取得・dHash 変化検知・解析タスクのスケジューリング
  （既存 `AnalysisScheduler` / asyncio パターンを踏襲）。設定値（間隔・上限・しきい値・フラグ）を
  `config.py` に追加。
- **コスト**: 上限 60 枚 × vision 呼び出しがセッションあたりの最悪コスト。メトリクスで実測して
  既定値を調整する（指標ハックではなく実測ベースで）。
- **観測性（原則3）**: キーフレーム採択/棄却のメトリクス `sanba_screen_keyframes_total{decision}`、
  解析 span、Langfuse トレース。「Live が映像をどう扱ったか」は観測できないままなので、
  こちらの経路が事実上の観測点になる。
- **テスト**: 単体（変化検知・ガード条件・重複抑制）、結合（フェイク映像トラック → 解析 → grounding 投入）、
  E2E（Playwright で画面共有 → 要件候補が UI に出る）。
- **ADR-0004 改訂**: 「共有中のみ有効化」の宿題を本 ADR で引き取ったことを追記する。
- 本 ADR は提案中。最終判断は人間レビューを経て Accepted とする（CLAUDE.md 原則1）。
