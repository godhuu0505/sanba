## コンテキスト

画面文言の洗い出し（`ui-copy-inventory.md` / #405）で、UI コピーに次の3課題が確認された。

1. **方針と実装の乖離**: ユビキタス言語は「**正本は機能名**。産婆・問答・オーレ！等の古語/擬人化は"デモ演出"であって
   要件・コピーの語ではない」（`ubiquitous-language.md §0`）と定めるが、実装コピーは古語・擬人化が多用され
   （`問答`／`見極めの刻にございます`／`〜ませぬ`／`オーレ！ 要件、産まれました`／`絵巻`／`検める`／`奉る`）、
   方針と実装が乖離している。初見の利用者には意味が取りにくい。
2. **表記のゆれ**: 同一概念に複数の語が併存する（開始行為＝壁打ち/要件サンバ/深掘り、役割＝利用者/顧客/PdM、
   参考資料/素材/資料、連携/前提リポジトリ、未確定/未解消 など）。
3. **専門語の補足不足**: `索引`／`URL キーワード`／`MoSCoW`／`確信度`／`言葉×画の矛盾` などジャーゴンに説明がなく、
   補足表示の UI 部品（ツールチップ/ポップオーバー）も実装に無い（依存は `@radix-ui/react-slot` のみ）。

**トーンの二重化について（本 ADR の主眼の改訂）**: 当初は 2 トーン（通常＝企画/開発者向け ↔ end_user＝利用者向け）で
UI 文言を二重に持つ設計だった（ADR-0032 の「UI 文言のモード別切替」）。しかしレビューで
**「文言はできる限り単一化したい」**と決定した。ADR-0032 が要求するのは「利用者に開発語彙（MoSCoW・非機能 等）を
露出しない」ことであり、これは **UI 文言を単一化しつつ平易側に寄せれば満たせる**（開発語彙を出さない＝むしろ強化）。
したがって **UI 文言のモード分岐は廃し、精密さはヘルプ（?）で補う**。ADR-0032 の end_user モード自体
（エージェントが利用者の語彙で問う・grounding 出力制御）は**実行時挙動として維持**する。

## 決定

1. **演出語（古語・擬人化・感嘆）を機能UIコピーから排除し、機能名・平叙体に統一する。**
   産婆術の世界観は**ブランド説明・マーケ・`/design` カタログ**の演出としてのみ残す。
2. **UI 文言はモードで分岐せず「単一文言」に統一し、平易側に寄せる。** 専門的な正確さ（MoSCoW・確信度の定義・索引 等）
   はラベルに載せず **HelpIcon（?）で補う**。
   - ADR-0032 の end_user モードは **エージェントの実行時挙動**（プロンプトの語彙・product 語彙シード・
     grounding 出力制御＝ADR-0032 決定7/8）としてのみ維持し、**静的 UI 文言は分岐しない**。
   - ADR-0032 決定7「開発語彙を UI に露出しない」は、単一文言を**平易側に統一**することで満たす
     （例: 分類ラベルから `非機能` を外し `使い心地` に、優先度は `MoSCoW/Must` を出さず `ぜひ必要…` にし詳細はヘルプ）。
3. **用語の単一の正はユビキタス言語ハブ。** 本 ADR 付録A（統一用語辞書）と付録B（画面別詳細）を確定し、
   `docs/reference/ubiquitous-language.md` に「UI 用語統一表」として反映する。実装は常にそれを参照する（ADR-0050 原則7）。
4. **ヘルプ補足はデザインシステム部品 `HelpIcon`（Radix Popover）に統一する。** ラベル直後の (?) をタップ/クリックで開く。
   依存 `@radix-ui/react-popover` を追加（モバイル一本道が正本のため hover ツールチップは不採用）。a11y 要件は下記。
5. **再ゆれ防止の検査を CI に入れる。** 「禁止バリアント→正語」表で `apps/web/{app,components}` に禁止語
   （`問答`・`要件絵巻`・`要件サンバ`・`壁打ちを始める`・`〜ませぬ`・`矛盾`検知ラベル 等）が再混入しないことをテストで検査
   （`docs/**`・ADR・glossary・`/design`・テスト自身は対象外）。
6. **スコープをセグメント（S0〜S7）に分割し issue 化、段階的に PR する。** S0 が依存の起点。

## ヘルプ UI（`HelpIcon`）の設計

- **依存追加**: `@radix-ui/react-popover`。
- **部品**: `apps/web/components/sanba/HelpIcon.tsx`。トリガは小さな (?) ボタン、内容は Popover。
  props（案）: `term`（索引キー・aria）／`title`／`children`（本文）。本文は**ハードコードせず** `lib/help/`
  （用語→`{title, body}` の**単一辞書**）から引く＝ヘルプ文の単一の正。
- **アクセシビリティ（必須）**: `aria-label="{term}の説明"`、キーボード開閉、`Esc` で閉じる、Popover の role/フォーカス管理、
  色のみに依存しない、モバイルで画面外に出ない衝突回避。ADR-0033 の a11y 方針（`supplements/0033-a11y-contrast.md`）に反しない。
- **観測性（任意・CLAUDE.md 原則3）**: ヘルプ開封を軽量イベントで記録し、分かりにくい用語を可視化できるとよい。

## 再ゆれ防止

- `apps/web` に「禁止バリアント→正語」対応表を持つ**テスト**（ソース走査）を追加し、禁止語の再混入を落とす。
- 許可リスト（`docs/**`・ADR・glossary・`/design`・テスト自身）は除外。出典は本 ADR 付録A/B と glossary の「UI 用語統一表」。

### 実装時に確認が残る項目

- **選択肢の観点 `どう並ぶ？`（`how`）** の意味を実装で確認し、名詞ラベル（暫定「位置づけ」）を確定する。
- **backend 由来の露出文字列**（`packages/sanba_shared/.../output_formats.py` の既定テンプレの `絵巻`/`深掘り`/`セッション`）は
  web でなく共有パッケージの修正。S6 と別 PR で対応する。
- **`ChatBubble` の既定グリフ `産`/`企`** の機能中立化は任意（P2）。

## 検討したが採用しなかった選択肢

- **UI 文言をモード別に二重管理（当初案）**: 却下。保守コストと表記分岐を生む。**単一文言＋ヘルプ**で、利用者保護
  （ADR-0032：開発語彙を出さない）と分かりやすさを両立できる。モードはエージェント実行時挙動として残せば足りる。
- **開発語彙側に統一**: 却下。ADR-0032 が利用者への MoSCoW・非機能 等の露出を禁止。よって**平易側に統一**する。
- **全面 i18n（next-intl 等）導入**: 却下（現時点）。MVP には過剰。単一文言＋ヘルプ辞書で足り、将来別 ADR で検討。
- **hover 型ツールチップ**: 却下。モバイル一本道でタッチでは開きにくい。タップ式 Popover を採る。
- **詳細対応表を ADR 本文に内包**: 却下。量が多く、ADR-0050 原則5 に従い `supplements/0054-copy-and-help-map.md` に分離。
