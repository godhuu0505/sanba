# ADR-0046: 音声リアルタイム worker から要件分析（ADK 多段チェーン）を分離する

- ステータス: Proposed
- 日付: 2026-07-07
- 関連: ADR-0002（マルチエージェント・トポロジ）/ ADR-0037（バックグラウンド先読みと注入ポリシー）/
  ADR-0038（音声ターン検出とセッション復旧）/ ADR-0039（音声入力精度）

## コンテキスト

本番セッション `sess-2d51da04` の会話破綻調査で、リアルタイム音声処理と要件分析が
**同一プロセス・同一イベントループに同居している**ことの構造的リスクが表面化した。

- 要件分析は `interview_lead` + 3 サブエージェント（nfr / scope / contradiction）を
  `gemini-2.5-flash` で**逐次**呼ぶ多段チェーン（AFC 最大 10 往復）で、最も高価な LLM 経路。
  当該セッションでは背景分析が **30 秒でタイムアウト**し、`Root node interview_lead was
  cancelled` に至った。
- ADR-0037 は分析をツール同期実行から切り離して**バックグラウンド化**したが、その実体は
  なお音声 worker の **同一イベントループ上での `await`** である（grounding 検索・先読みは
  `asyncio.to_thread` へ逃がしているのに、分析経路だけ逃がしていない非対称がある）。
- さらに `analyze_requirements` の相乗り待ち（走行中の背景分析へ相乗りして `await` する）は、
  モデルがこのツールを呼ぶと **最大 30 秒 音声ターンをブロック**し得るランドマインになっている。
- LiveKit Agents はジョブ単位でサブプロセスを起こし、本番は `cpu=2`。レイテンシ最優先の
  リアルタイム音声と、スループット志向のバッチ的 LLM オーケストレーションを 1 プロセスに
  同居させると、**失敗ドメインと CPU/イベントループを共有**してしまう。

なお、直近の修正（本 PR の P0）で書き込み系 I/O のループ・オフロードと無駄な埋め込み呼び出しの
除去は済んでいるが、**分析経路そのものの分離**は未着手であり、それを本 ADR で扱う。

## 決定（提案）

要件分析（ADK チーム）を音声 worker から**別の失敗ドメイン**へ切り出す。会話への
非同期割り込みをしない ADR-0037 決定1 は維持し、分析結果の届け先は従来どおり
**(1) 次のツール返答、(2) web の検知カード**の 2 経路に限る。

段階的移行を提案する:

- **段階1（短期・タイムアウト境界は実装済み）**: 音声ターンを分析で塞がないよう、2 段の上限で守る。
  - 背景分析 1 回の上限（`analysis_timeout_seconds`）は 30→**20 秒**。健全時は 10 秒未満で完了する
    ため、完了しきれる余裕を残しつつ暴走を抑える。超過は fail-soft で破棄する。
  - `analyze_requirements`（ツール）が走行中の背景分析へ相乗りして待つ上限
    （`analysis_ride_along_timeout_seconds`）を **8 秒**に固定。超過したら音声ターンをそれ以上塞がず、
    直近結果（無ければヒューリスティック `heuristic_result`）を即返す。背景は走り切り、結果は次ターンに
    反映される（競合する同期分析は起動しない＝二重往復と更なる沈黙を防ぐ）。音声ターン保護の主レバー。
  - **残**: `analyze_transcript`（ADK `run_async`）自体を専用ループのワーカースレッドへ丸ごと逃がして
    音声ループのジッタを消す件は、実 ADK をスレッドで回す挙動の検証が要るため未了。段階2 の
    サービス切り出しで代替される可能性もあり、そちらと合わせて判断する。
- **段階2（目標形・要レビュー）**: 分析を**独立した実行単位**（専用 Cloud Run サービス、
  または `apps/api` 内の分析ワーカー）へ切り出し、音声 worker からは非同期トリガー
  （Pub/Sub もしくは内部 HTTP）で起動する。結果は Firestore の検知として永続化され、
  既存の data channel（検知カード）で web へ届く。音声 worker の責務は「聴く・話す・
  軽い記録・publish」に縮小する。

## 理由 / 検討した代替案

- **A. 独立サービス化（段階2 の本命）**: 失敗ドメイン・CPU・スケールを音声から分離でき、
  分析の遅延・失敗が音声ターンに波及しない。反面、インフラ追加（サービス / Pub/Sub / IAM /
  Secret Manager）と往復レイテンシ、transcript 受け渡しと検知の結果整合の設計が要る。
- **B. 同一プロセスでスレッド/プロセスプールへオフロード + ハードタイムアウト（段階1 の暫定）**:
  インフラ変更が最小で早い。ただし CPU とプロセスは共有のままで、隔離は部分的。
- **C. 現状維持（イベントループ上で分析）**: 不採用。失敗ドメイン共有と 30 秒ハングの
  リスクが残る。少なくとも相乗り待ちの無制限ブロックは塞ぐ必要がある。

段階1 で即効性のあるリスク低減を行い、段階2 の独立サービス化は費用対効果とインフラ
レビューを経てから確定する、という順序を推奨する（断定はせず、本 ADR では方向づけに留める）。

## 影響 / フォローアップ

- **観測性（原則3）**: 本 PR で分析レイテンシ（`analysis` の `duration_ms` + span）を一級化済み。
  段階2 では境界をまたぐトレース文脈の伝播、キュー滞留・失敗数・スキップ数のメトリクスを追加する。
- **IaC**: 段階2 が独立サービスなら、Cloud Run サービス・Pub/Sub トピック・IAM・Vertex AI
  アクセス（Secret Manager / WIF）の Terraform を追加する。`infra/` 変更は必ずレビューする。
- **テスト**: 分析境界の契約テスト（入力＝transcript、出力＝検知/次の一問）と、分析が遅い/
  不達でも音声 worker が heuristic フォールバック（`analyze_transcript` の既存の劣化経路）で
  会話を止めないことの結合テスト。
- **移行**: 段階1→段階2 は段階的。段階1 だけでも「分析の遅延が音声を塞がない」を満たせる。
- **受け入れ基準（指標をハックしない）**: 分析経路の遅延・失敗が**音声ターンの p95 レイテンシに
  影響しない**ことを計測で示す。目標値は段階1 のメトリクスを見てから確定する。
- **セキュリティ / 隔離**: 分析に渡す transcript は PII マスク方針（issue #10）と
  セッション隔離（ADR-0028）を境界越えでも維持する。
