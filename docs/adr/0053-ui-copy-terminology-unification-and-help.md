## コンテキスト

画面文言の洗い出し（`ui-copy-inventory.md` / #405）で、UI コピーに次の3課題が確認された。

1. **方針と実装の乖離**: ユビキタス言語は「**正本は機能名**。産婆・問答・オーレ！等の古語/擬人化は"デモ演出"であって
   要件・コピーの語ではない」（`ubiquitous-language.md §0`）と定めるが、実装コピーは古語・擬人化が多用され
   （`問答`／`見極めの刻にございます`／`〜ませぬ`／`オーレ！ 要件、産まれました`／`絵巻`／`検める`／`奉る`）、
   方針と実装が乖離している。初見の利用者には意味が取りにくい。
2. **表記のゆれ**: 同一概念に複数の語が併存する。代表例:
   - セッション開始の行為＝`壁打ちを始める`（ホーム）／`要件サンバを始める`（準備）／`深掘りを開始する`（参加）
   - 役割＝`利用者/企画者/開発者`（`AUDIENCE_LABELS`）／`企画(PdM)/顧客`（参加）／`企画(PdM)/エンジニア/顧客`（glossary）
   - 参考資料＝`参考資料`／`素材`／`資料` ／ 前提リポジトリ＝`連携リポジトリ`／`前提リポジトリ`
   - 未解消の件数＝`未確定`（会話ミニ状況）／`未解消`（終了確認・判定）
3. **専門語の補足不足**: `索引`／`URL キーワード(slug)`／`MoSCoW`／`確信度`／`言葉×画の矛盾`／`前提リポジトリ`
   などジャーゴンに説明が無く、特にオーナー向け管理画面と会話中の可視化で初見に不親切。加えて、
   **補足表示のための UI 部品（ツールチップ/ポップオーバー）が実装に存在しない**（依存は `@radix-ui/react-slot` のみ）。

SANBA は 2 トーン設計（**通常**＝企画/開発者向け ↔ **end_user**＝利用者向け／ADR-0032）を持つため、
「統一」は *全画面で1語* ではなく「1概念につき〈通常の正語〉と〈利用者向けの正語〉をそれぞれ1つに定める」形になる。

## 決定

1. **全モードで機能名に統一し、演出語（古語・擬人化・感嘆）を機能UIコピーから排除する。**
   ユビキタス言語の「正本は機能名」に実装コピーを寄せる。end_user モードはさらに平易化（漢語→和語・専門語回避）
   するが、意味は機能名と一致させる。産婆術の世界観は**ブランド説明・マーケ・`/design` カタログ**の演出として残しうるが、
   操作を担う機能UIの文言からは外す。
2. **用語の単一の正はユビキタス言語ハブ。** 本 ADR で全画面のコピー正語・ヘルプ本文を確定し（付録A＝横断用語辞書、
   付録B＝画面セグメント別の詳細対応表＝supplement）、`docs/reference/ubiquitous-language.md` に「UI 用語統一表」として
   反映する。実装は常にそれを参照する（ADR-0050 原則7）。
3. **ヘルプ補足はデザインシステム部品 `HelpIcon`（Radix Popover）に統一する。** 用語ラベルの直後の (?) を
   タップ/クリックで補足を開く。依存 `@radix-ui/react-popover` を追加する（モバイル一本道が正本のため hover 型
   ツールチップは採らない）。a11y 要件（下記）を必須とする。
4. **再ゆれ防止の検査を CI に入れる。** 「禁止バリアント→正語」表に基づき、`apps/web/{app,components}` に
   禁止語（例: `問答`・`要件絵巻`・`要件サンバ`・`壁打ちを始める`・`〜ませぬ`）が再混入していないことをテストで検査する
   （`docs/**`・ADR・glossary・`/design` カタログ・テスト自身は対象外）。web の CI ジョブで実行。
5. **スコープをセグメントに分割して issue 化し、段階的に PR する。** 依存の起点は基盤 issue（`HelpIcon`＋
   glossary 反映＋共有ラベル定数の更新＋禁止語テスト）。以降、画面セグメント（付録B の S1〜S7）を順に反映する。

## ヘルプ UI（`HelpIcon`）の設計

- **依存追加**: `@radix-ui/react-popover`（既存 shadcn/Radix 系と同系統）。
- **部品**: `apps/web/components/sanba/HelpIcon.tsx`。トリガは小さな (?) ボタン、内容は Popover。
  - props（案）: `term`（何の補足か。aria とヘルプ本文の索引キー）／`title`／`children`（本文）。
  - 本文は**ハードコードせず** `lib/help/`（用語→{title, body} の辞書。通常/end_user 出し分け）から引く＝ヘルプ文の単一の正。
- **アクセシビリティ（必須）**: トリガに `aria-label="{term}の説明"`、キーボードで開閉可、`Esc` で閉じる、
  Popover に適切な role/フォーカス管理、色のみに依存しない、モバイルで画面外に出ない衝突回避配置。
  ADR-0033 の a11y 方針（`supplements/0033-a11y-contrast.md`）に反しないこと。
- **観測性（任意・CLAUDE.md 原則3）**: ヘルプ開封を軽量イベントで記録し、どの用語が分かりにくいかを可視化できるとよい。

## 再ゆれ防止

- `apps/web` に「禁止バリアント→正語」対応表を持つ**テスト**（ソース走査）を追加し、禁止語の再混入を落とす。
- 誤検出回避のため、許可リスト（`docs/**`・ADR・glossary・`/design` カタログ・テスト自身）は除外する。
- 対応表は本 ADR 付録A/B と `ubiquitous-language.md` の「UI 用語統一表」を出典とする。

## 影響 / 移行（イシュー分割）

エピック 1 本＋以下の子 issue に分割する（各 issue は本 ADR と付録B の該当セグメントを参照）。**S0 が依存の起点**。

- **S0 基盤**: `HelpIcon`（Radix Popover）＋ `lib/help/` ヘルプ辞書 ＋ ユビキタス言語への「UI 用語統一表」反映
  ＋ 共有ラベル定数（`lib/realtime/mapping.ts`・`lib/audience.ts`・各 `STATUS_LABEL`）の正語化 ＋ 禁止語テスト。
- **S1 入口フロー**: `EntryFlow`（ホーム/準備）・`login`・`ConversationStart`（会話開始）。
- **S2 会話・選択肢・検知**: `ConversationShell`/`ChatHistory`/`BottomBar`/音声状態/選択肢シート/検知ピン ほか。
- **S3 参考資料**: `MaterialsList`/`MaterialDetailSheet`/`MaterialSourceSheet`/`MaterialCancelDialog`。
- **S4 要件一覧・深掘り・判定**: `RequirementsTab`/`RequirementsScrollList`/`DeepDiveList`/`RequirementCard`/`JudgmentGate`。
- **S5 結果・過去の要件一覧**: `ResultView`/`results/[id]`/`results`/`SessionHistoryList`。
- **S6 アプリ管理**: `products`/`products/[id]`/`Product*Card`（5枚）。
- **S7 招待・参加・システム画面・設定・ナビ**: `join`/`member-invites`/`MemberInviteNotices`/`error`/`not-found`/
  `AccessErrorScreen`/`settings`/`GitHubLinkCard`/`AppShell`/`SideMenu`/`AccountMenu`。
- 各セグメントの PR では、日本語文言をアサートしている既存テストも正語に合わせて更新する（テストがコピーを固定する）。

### 実装前に人間の最終確認が要る決定（付録Bで詳述）

機械的でなく判断を伴う・影響範囲が広い項目。レビューで確定する。

- **開始行為の呼称**を `要件インタビューを始める`（end_user: `聞き取りを始める`）に統一し、`壁打ち`/`要件サンバ`/`深掘り` を
  UI 表層から排除する（概念語としてはドキュメントに残す）。ブランドの手触りに関わるため要確認。
- **機能カード名の改名** `深掘りリンク → 要件インタビューのリンク`（`ProductInvitesCard`）は露出箇所が多い。
- **`未確定 → 未解消`** の統一（会話ミニ状況を判定/終了確認に揃える）。
- **選択肢の観点 `どう並ぶ？`（`how`）** の意味を実装と確認し、名詞ラベル（暫定「位置づけ」）を確定する。
- **backend 由来の露出文字列**（`packages/sanba_shared/.../output_formats.py` の既定テンプレの `絵巻`/`深掘り`）は
  web でなく共有パッケージの修正。S6 と別 PR で共有パッケージ担当と調整。
- **`ChatBubble` の既定グリフ `産`/`企`**（産婆/企画の暗示）を機能中立にするかは任意（P2）。

## 検討したが採用しなかった選択肢

- **全面 i18n（next-intl 等）導入**: 却下（現時点）。MVP には過剰。まず機能名統一＋ヘルプ辞書の部分集約で足り、
  将来多言語が必要になれば別 ADR で導入する。
- **演出語の全面維持**: 却下。わかりやすさとユビキタス言語の「正本は機能名」方針に反する。世界観は演出領域に限定する。
- **hover 型ツールチップ**: 却下。モバイル一本道フローが正本でタッチでは開きにくい。タップ式 Popover を採る。
- **用語表を ADR に載せず glossary だけで管理**: 却下。決定の経緯・却下案・移行計画は ADR に残すのが本リポジトリの規律
  （ADR-0050）。glossary は「現行の正」を保持し、ADR は「なぜそう決めたか」を保持する二層で運用する。
- **詳細対応表を ADR 本文に内包**: 却下。量が多く可読性を損なうため、ADR-0050 原則5 に従い付随資料を
  `supplements/0053-copy-and-help-map.md` に分離し、本 ADR から参照する。

## 付録A — 横断用語辞書（正語）

複数画面で再利用される中核語。ここを直すと波及が広い。

| 概念 | 正語(通常) | 正語(end_user) | 統一の要点 |
|---|---|---|---|
| セッション開始の行為 | 要件インタビューを始める | 聞き取りを始める | `壁打ちを始める`/`要件サンバを始める`/`深掘りを開始する` を統一。壁打ち/要件サンバ/深掘りは UI 表層から排除（概念語としてはドキュメントに残す） |
| 会話（画面・行為） | 会話 | 会話 | `問答` を廃止 |
| 要件の一覧・記録 | 要件一覧 | 要件一覧 | `要件絵巻`/`過去の要件` を統一（画面見出しは「過去の要件一覧」可） |
| 参考資料 | 参考資料 | 参考資料 | `素材`/`資料` を「参考資料」に統一（写真・録画と並ぶ“文書”の意味の「資料」は「文書」に。付録B S3 参照） |
| 前提リポジトリ | 前提リポジトリ | 前提リポジトリ | `連携リポジトリ` を統一 |
| 役割 | 利用者／企画者／開発者 | 同左 | `企画(PdM)`/`エンジニア`/`顧客` 等の別表記を排除。参加時 roleLabel・glossary もこれに合わせる |
| 検知(矛盾/抜け/不明瞭) | 矛盾／抜け／不明瞭 | 食い違い／確認したい点／あいまい | 現行 `mapping.ts` を維持（既に機能名） |
| 要件カテゴリ | 機能/非機能/制約/境界/未解決 | 機能/使い心地/前提/範囲/確認中 | 現行維持 |
| MoSCoW | Must 必須 / Should 望ましい / Could できれば / Won't 今回は対象外 | ぜひ必要 / あると助かる / できれば / 今回は見送り | 現行維持 |
| 未解消（の件数） | 未解消 | 未解消（会話中は「確認したいこと」の語と併用） | 会話ミニ状況の `未確定` を `未解消` に統一 |
| 確定/暫定（要件） | 確定要件 / 暫定要件 | 確認できた内容 / 途中までの内容 | end_user は平易化 |
| 確信度 | 確信度（高/中/低） | 確からしさ（高/中/低） | end_user 平易化。ヘルプで補足 |
| 出所 | 出所 | だれの発言か | end_user 平易化 |
| 索引（indexing） | 索引 | （利用者画面には出さない） | 語は維持しヘルプで補足 |
| 個人情報の秘匿 | 伏せ字にします | 伏せ字にします | `マスクします` を「伏せ字にします」に統一 |

※プロダクト名「SANBA」は残す。排除するのは行為の愛称「要件サンバ」。
