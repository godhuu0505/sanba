# ADR-0039: 対象アプリの選択を 01 ホームの開始ゲートにする

- ステータス: Accepted
- 日付: 2026-07-06

## コンテキスト
これまで対象のプロダクト・アプリ（ADR-0031）は 02 セッション準備のフォーム内で
選択していた。しかし壁打ちは常に「どのアプリの要件か」が起点であり、準備フォームの
中段に必須セレクトが埋まっていると、

1. ゴールや役割を書き進めた後に「アプリ未選択で開始できない」と気づく手戻りが起きる。
2. 準備画面のフィールドが多く、必須項目（役割・ゴール・同意）への集中が削がれる。

という体験上の問題があった。ユーザー要望（2026-07）としても「トップ画面でまず
アプリを選び、選べたら壁打ち開始ボタンが活性化する」流れが求められた。

## 決定
1. **アプリ選択は 01 ホームの開始ゲートにする。**
   ホームのヒーローカード内に「対象のプロダクト・アプリ」セレクト（必須）を置き、
   選択できるまで「＋ 壁打ちを始める」CTA を活性化しない。
   - 候補が 1 件なら自動選択する（既存の突き合わせ effect を踏襲）。
   - 候補が 0 件ならセレクトを無効化し、アプリ管理（/products）からの登録を案内する。
2. **02 セッション準備から選択 UI を外し、確定済みアプリ名の表示に置き換える。**
   準備画面の先頭に「対象のアプリ: ○○」の行を常時表示し、どのアプリで準備を
   進めているかを見失わせない。開始条件（selectedProduct の実在）は従来どおり
   fail-closed を維持する。
3. **アプリ未確定のまま 02 に居着けないようにする。**
   /prepare 直リンク・stale な保存値のクリア後など、候補 settle 後もアプリ未確定の
   場合は 01 ホームへ戻して選び直させる（02 に選択 UI が無いため、留まると開始が
   永久に塞がる）。
4. **候補取得（GET /api/products/mine）はログイン解決後に step を問わず行う。**
   従来は 02 に入るまで遅延させていたが、ホームがゲートになったため取得を前倒す。
   選択値の sessionStorage 永続（#179 / prepFormStorage.productId）は従来どおりで、
   ホーム→準備→/login 往復でも選択は保たれる。

## 影響
- apps/web `EntryFlow.tsx`: ホームにセレクト＋CTA ゲート、準備は表示のみ＋未確定時の
  巻き戻し。準備画面の「対象のプロダクト・アプリの選択が必要です。」注記は不要になり削除。
  巻き戻しは構造化ログで観測できるようにする（CLAUDE.md 原則3 / 配線は #232）。
- 既知のトレードオフ: 02 準備で入力中に対象アプリが（別セッション等で）削除されると、
  candidates 再取得のタイミングでホームへ巻き戻される。入力値は sessionStorage に残る
  ためデータ損失はないが、体験としては唐突。想定頻度が低いため許容し、頻発するようなら
  巻き戻しログを根拠に警告表示等を検討する。
- GitHub リポジトリ候補（ADR-0027）の遅延取得（02 に入るまで叩かない）は不変。
- API・agent への変更はない（createSession への product_id 送信・文脈シードは従来どおり）。

## 代替案
- **準備画面に選択を残したままホームにも置く**: 選択箇所が二重になり、どちらが正か
  分かりにくい。却下。
- **アプリ未確定の /prepare 直リンクで選択モーダルを出す**: 一本道（ADR-0017）に
  反する分岐 UI が増える。ホームへ戻すほうが単純で、選択済みなら直リンクは従来どおり
  機能する。却下。
