## エグゼクティブサマリー
 
本ハッカソンは**Findy主催・Google Cloud Japan協賛**の新規ハッカソンですが、Google Cloud協賛のAIエージェント系ハッカソン（Zenn主催の第1〜4回 AI Agent Hackathon、ADK Hackathon）と**審査体制・必須技術・評価軸がほぼ同じ系譜**にあります。過去の公表審査基準と受賞作品の傾向を統合すると、勝つために重要なのは次の3点に集約されます。
 
1. **「技術力」より「課題発見と解決策の有効性」**が決定的に重要

## 2. 審査基準の徹底分析
 
### 2.1 公式に示唆されている評価軸
 
Findyの公式情報・Google Cloud公式ブログから、本ハッカソンの**3つの動詞**が評価の骨格になっていることが明示されています。
 
#### ① つくる
> 「アイデアの面白さだけでなく、エージェントとしての必然性、自律的に判断しタスクを実行する設計まで踏み込んで評価します」
 
→ **キーワード: 「エージェントとしての必然性」「自律性」**
→ 単なるLLMチャットボットや、人間が指示を順番に出すだけのワークフローは評価されない。**自律的に意思決定・行動するAgenticな設計**が必須。
 
#### ② まわす
> 「CI/CDなどDevOpsのフローを構築し、AIを継続的に改善するサイクルを参加者自身が回します」
 
→ **キーワード: 「CI/CD」「継続的改善」**
→ 本ハッカソン最大の差別化ポイント。**他のAI Agentハッカソンには無い「DevOps」という冠**がある以上、ここが弱いと評価されない。
 
#### ③ とどける
> 「Google Cloudへのデプロイを通じて、スケーラブルな環境で本番品質のプロダクトをユーザーに届けます。動くものをつくる、ではなく、届くものをつくる」
 
→ **キーワード: 「本番品質」「スケーラブル」**
→ 動くデモではなく、実運用に耐えうるレベルの完成度が求められる。
 
### 2.2 姉妹ハッカソンで明文化されていた3つの審査基準
 
同じGoogle Cloud協賛の第4回Agentic AI Hackathon（Zenn主催）では、公式に審査基準が明示されていました。**本ハッカソンも同等の基準で採点される可能性が極めて高い**と推定されます。
 
| 審査基準 | 内容 |
|----------|------|
| **【課題の新規性】** | 多くの人が抱えていて、いまだに解決策が与えられていない課題を発見したプロジェクトを評価 |
| **【解決策の有効性】** | 提案されたソリューションがその中心となる課題に効果的に対処し、解決しているかを評価 |
| **【実装品質と拡張性】** | アイデアをどの程度実現し、必要なツールを活用し、拡張性・運用しやすさ・費用対効果の高いソリューションを作成できたかを評価 |
 
第4回ハッカソンの「奨励賞（5枠）」は「**課題発見・アイデア・実装のどこかが光っていれば受賞**」とされており、本ハッカソンの「特別賞10万円×6組」も同様の運用と推測されます。
 
### 2.3 統合: 本ハッカソンの推定スコアリング構造
 
過去のハッカソンの評価軸と、本ハッカソン固有の「DevOps」要素を統合すると、おおむね以下のような採点構造になると予想されます。
 
```
【最優秀賞】= 全項目で高得点 (バランス型の完成度勝負)
  ├─ 課題の新規性・社会的インパクト
  ├─ 解決策の有効性（エージェントの必然性）
  ├─ 実装品質・拡張性
  ├─ DevOpsサイクル（CI/CD・継続的改善）★本ハッカソン固有
  └─ プレゼンテーション能力
 
【優秀賞 (×3)】= 最優秀賞に迫る完成度
【特別賞 (×6)】= 上記いずれかが突出
```

## 5. 優勝・入賞のための実践戦略
 
### 5.1 戦略①: 審査基準逆算でコンセプトを作る
 
過去の受賞者（Zenn AI Agent HackathonでTech Deep Dive賞受賞者の事例）が実践した、**AIによる審査基準逆算法**が極めて有効です。
 
**手順:**
1. 本レポートの審査基準（課題の新規性／解決策の有効性／実装品質と拡張性／DevOps）をAIにインプット
2. 「審査基準を十分に達成できるコンセプト」をAIと対話して30案出す
3. 採用技術（Gemini Enterprise Agent Platform、ADK、Cloud Run等）を**どれだけ盛り込めるか**を考える
4. 最後にAIに「受賞の可能性が十分にあるか」評価させて選定
### 5.2 戦略②: 「DevOpsサイクル」を必ず実装する
 
**ここが本ハッカソン最大の勝ち筋**です。他のAIハッカソンとの最大の違いが「DevOps」だからです。「動くAIエージェントを作りました」では絶対に勝てません。
 
**実装すべき要素チェックリスト:**
 
- [ ] **CI/CD パイプライン**（GitHub Actions + Cloud Build 等）
- [ ] **Infrastructure as Code**（Terraform / gcloud config 等）
- [ ] **観測性 (Observability)**: Cloud Logging / Cloud Trace / Cloud Monitoring
- [ ] **LLMOps**: プロンプトのバージョン管理、評価データセット、回帰テスト
- [ ] **エージェントの評価サイクル**: 出力品質を継続的に測定・改善する仕組み
- [ ] **オートスケール**: Cloud Run の自動スケーリング設定
- [ ] **コスト管理**: 予算アラート、推論コストの可視化
→ **これらを「実装している」だけでなく、「なぜ必要か」「どんな効果があったか」をピッチで語れること**が決定的に重要。
 
### 5.3 戦略③: 「エージェントの必然性」を演出する
 
「AIエージェントである必要性」を強く打ち出します。
 
**避けるべきパターン:**
- ❌ チャットボット（人間が逐一指示する）
- ❌ 単純な1ステップ生成タスク（要約・翻訳など）
- ❌ Geminiに1回投げるだけで終わるアプリ
**目指すべきパターン:**
- ✅ **マルチステップの自律判断**: タスクを分解 → 計画 → 実行 → 検証 → 自己修正
- ✅ **マルチエージェント協調**: 専門エージェントが連携（Lead Agent + Sub Agents）
- ✅ **Tool Use**: 外部API・データベース・コードを自律的に呼び出す
- ✅ **記憶・学習**: 過去の対話やフィードバックを次の判断に活かす
→ **ADK（Agents Development Kit）**でマルチエージェント構成を組むのは、評価される確率が高い王道。
 
### 5.4 戦略④: 課題発見こそが最大の差別化
 
過去の受賞者が口を揃えて言うのは「**技術力よりアイデア・課題解決能力が重視される**」ということです。
 
**課題発見のフレームワーク:**
1. **自分・自分の身近な人が本当に困っていること**から出発する（リアリティの源泉）
2. それを**狭く深く**掘る（広い課題ほど誰でも思いつく）
3. その課題が**AIエージェントでなければ解けない理由**を明確にする
4. **3分のデモで「あぁ、確かに困るよね」と審査員に共感させられる**ストーリーを作る
**Findyのドメインを意識した着想例:**
- 開発生産性の可視化・改善を自律的に行うエージェント
- コードレビュー・ペアプロを自律的に補佐するエージェント
- インシデント対応・障害分析を自動化するSREエージェント
- 技術的負債の発見・リファクタ提案を継続的に行うエージェント
→ Findy主催であることを踏まえ、**「エンジニア向けDevOpsツール」「開発生産性領域」**は審査員の共感を得やすい可能性が高い。
 
### 5.5 戦略⑤: 提出物の質を最大化する
 
提出物は通常「**GitHubリポジトリ + デプロイURL + 紹介記事**」の3点セット。1次審査では記事とリポジトリしか見られない可能性が高いため、**記事・READMEで勝負が決まる**前提で準備します。
 
**記事に必ず含めるべき内容:**
1. **対象ユーザー・解決する課題**（最重要：審査基準の①と直結）
2. **「なぜAIエージェントなのか」の必然性の説明**
3. **システムアーキテクチャ図**（GCPサービスとデータフローを明示）
4. **DevOpsサイクル図**（CI/CD・LLMOps・観測性の全体像）
5. **3分以内のデモ動画**（Before/Afterや自律的な振る舞いが見える）
6. **継続改善のサイクルの説明**（「まわす」を実装した証拠）
7. **本番品質の根拠**（スケーラビリティ・コスト・セキュリティ）
### 5.6 戦略⑥: 公式リソースをフル活用する
 
参加者全員がベースラインを上げる仕組みが用意されているため、**使わない人は相対的に不利**になります。
 
**必ず参加すべきもの:**
- **Agentic AI Bootcamp 2026**（6/1〜6/12、Google Cloud主催、無料）
  - ADK・Gemini Enterprise Agent Platform・Cloud Run・Gemini API のハンズオン
  - 同じ技術スタックの参加者と差をつけるための基礎固め
- **チームビルディングイベント**（6/7、ファインディオフィス）
  - チーム参加の場合も、他チームのレベル感を測れる貴重な機会

## 6. リスクと注意点
 
### 6.1 技術選定のリスク
 
- **Gemini Enterprise Agent Platform**は新しい技術スタックのため、情報が少ない可能性。Bootcampでの習得が必須。
- **ADK** は急速に進化しているため、提出時点の最新バージョンに対応すること。
- Google Cloudクレジット配布は明示されていないため、**自己負担でのGCP利用コスト**を見積もる必要あり。
### 6.2 タイムマネジメント
 
申込開始4/27〜提出7/10の約**2.5ヶ月**しかありません。過去の受賞者には2週間で仕上げた例もあるが、DevOpsサイクル含む本ハッカソンは**最低でも実働4〜6週間**は必要と見るべき。
 
**推奨スケジュール:**
- 5月下旬: コンセプト確定・チーム結成
- 6月上旬: Bootcamp参加 + アーキテクチャ設計
- 6月中旬〜下旬: 集中実装期間
- 7月第1週: デプロイ・観測性・CI/CD仕上げ
- 7月10日締切まで: 記事執筆・デモ動画撮影
### 6.3 重複応募・参加条件
 
- 18歳以上、日本居住者
- 政府機関職員、スポンサー従業員等は参加不可
- 複数プロジェクト参加可否は本ハッカソン固有のルール確認が必要
