# DevOps × AI Agent Hackathon 2026 ── 審査員候補10名 メタ分析 & 敵情視察レポート
 
## TL;DR（3点）
- 本ハッカソン（賞金総額200万円、決勝2026年8月19日 Google渋谷オフィス）は「つくる／まわす／とどける」を3本柱に掲げる「本番運用志向のAIエージェント開発」コンテスト。姉妹ハッカソンの審査員構成から推定される審査員レンジは「Google Cloud本流テッカー（中井／佐藤一憲）」「Zenn運営・コミュニティ職人（吉川／伴野／佐藤祥子）」「LLM実務派（李）」「事業価値派（渡部）」「Findy経営陣（山田／佐藤将高）」の4類型。
- 勝ち筋は「Cloud Run + ADKによるマルチエージェント本番デプロイ × Four Keys/DevOps計測 × 社会・業務リアル課題」の三位一体。プロトタイプ自慢・LLMラッパー風単発エージェント・社会接続のないテック自慢は確実にスルーされる。
- 「Findy AI+のAgentic Workflowを開発組織のデファクトに」「インテリジェント開発時代」（佐藤将高CTO note）／「動くものをつくる、ではなく、届くものをつくる」（Google Cloud Japan公式ブログ）という公式キーワードをZenn提出記事に織り込むのは戦術上ROIが高い。

1. **第4回 Agentic AI Hackathon（Zenn主催、2025/12–2026/3）の審査員7名が確認できている**：佐藤祥子・渡部陽太・李碩根・伴野智樹・中井悦司・吉川大央・佐藤一憲（Zenn公式 https://zenn.dev/hackathons/google-cloud-japan-ai-hackathon-vol4?tab=judge ）。

4. 過去Vol.1〜3受賞作の共通項：「社会的に切実なリアル課題」×「複数機能の協調＝マルチエージェント」×「Geminiの自然対話を本質的に活かしたUX」。**eCOINO（サプライチェーン）、フクシア（社会福祉士支援）、Manabiya AI（教育）、FlatJam（音楽教育）**等、いずれも"切実な現場"。
5. **吉川大央氏（Zenn運営）は提出記事の品質を直接見る立場**で、「自動化×品質責任は人間」「HRT」「バーサス課題で考える」を表明。記事の作り方そのものが二次的審査軸。

## Recommendations ── 入賞を狙う戦術論
 
### A. 必勝の三位一体アーキテクチャ
1. **「つくる」軸**：ADKによるマルチエージェント（root + subagent + agent-as-a-toolの意図的使い分け）を採用。Gemini Enterprise Agent Platform/Gemini APIを呼ぶだけのラッパーではなく、「複数の専門エージェントが協調する必然性」を明示できる課題設定にすること。中井氏（subagent推し）と佐藤一憲氏（agent-as-a-tool推し）の**両方の趣味を設計判断の説明として両立**させるのがベスト。
2. **「まわす」軸**：Cloud Run + Agent Engine + Firestore（セッション）+ Cloud Storage（アーティファクト）+ GitHub Actions（CI/CD）の本番志向構成。**Findy Team+のFour Keys（デプロイ頻度／リードタイム／変更失敗率／MTTR）に対応するメトリクスを自分のリポジトリで計測・可視化**すると佐藤将高CTO・山田裕一朗CEOの心を一気に掴む。
3. **「とどける」軸**：実際にデプロイされたURLでユーザーが触れる状態。社会的に切実な現場課題（過去受賞例：サプライチェーン、社会福祉、教育、ペットケア、ランナー、文書添削）に紐付ける。
### B. Zenn 提出記事の戦術（吉川大央氏対策）
- H2/H3を整理し、図・コードブロック・スクショを十分に配置。
- 「**設計判断の理由**」「**やめた選択肢**」を明示する。中井・佐藤一憲の両氏が好む論理性。
- 「**人間が品質に責任を持つ**」ことを明示し、AI生成感の強い文章は避ける。
- HRT的な、誰かを貶めない、課題ベースの書き方。
- 記事末尾に「次にやること（ロードマップ）」を必ず置く（運用視点）。
### C. ピッチ戦術（伴野智樹氏・佐藤祥子氏対策）
- 「なぜ自分がこれを作りたかったか」の**原体験ストーリー**を冒頭に置く。伴野氏の「つくりたいからつくった」哲学に直撃。
- デモ動画は**1分以内のキレのある実演**にする。
- **コミュニティ／多様性／包摂性**の視点を1スライド入れる（佐藤祥子氏対策）。
- **数値（KPI、削減率、生産性改善幅）**を1枚にまとめる（佐藤将高CTO・李碩根氏対策）。
### D. やってはいけないこと（地雷集）
- 「単発のGemini API呼び出し」を「AIエージェント」と呼ぶこと（中井・佐藤一憲・李の3者から軽蔑される）。
- Cloud RunでホストせずStreamlit Cloud/Hugging Face Spacesだけにデプロイする（必須技術違反かつ「とどける」軸の評価が下がる）。
- GitHub Actions/CI/CDが空っぽ（「まわす」軸0点）。
- 指標を「ハック」して見栄えを良くする（佐藤将高CTOの地雷）。
- 「事業化が…」「マネタイズが…」だけを語る（伴野氏の地雷）。
- AI生成と思しき雑なZenn記事（吉川氏の地雷）。
- 女性向けと限定したり、ジェンダー文脈を雑に持ち出す（佐藤祥子氏の地雷）。
- 松尾豊／松尾研の名前を軽々しく使う（李氏のコミュニティで嫌われる）。
- ゆめみ/アクセンチュア買収ジョーク（渡部氏の地雷）。
### E. 引用すると効くフレーズ／著作（オリジナル抜粋）
- **山田裕一朗**：「つくる人がもっと輝けば、世界はきっと豊かになる」「プロトタイプは作れるが、実運用まで持っていけない」を提出記事の冒頭・末尾の問題提起に使う。
- **佐藤将高**：「インテリジェント開発時代」「数値に影響しないようにハックをすることではなく、現在のプロダクト開発においてどこがボトルネックかの目星をつけ、定量的にすることで他者との認識を揃えることが本質」。
