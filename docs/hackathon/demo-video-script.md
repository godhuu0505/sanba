> 審査の一次通過は「ProtoPedia ページ＋リポジトリ」で決まる（`judging-criteria-strategy.md §5.5`）。
> 動画で伝えるのは **「あぁ、確かに困るよね」への共感 → SANBA が自律的に動く様子 → 本番品質の証拠** の 3 点。

> 撮影は実 creds を繋いだ本番（または実 creds のローカル）で行う。手順は

## 撮影チェックリスト

- [ ] **実 creds で本物の経路**が通っていること（ヒューリスティック fallback ではない）。カット3 の音声応答、カット5 の ADK 分析、カット6 の Issue 起票が実物であること。
- [ ] カット3 は **barge-in（言いかぶせ）** が分かると強い（自然な会話性）。
- [ ] カット4 の矛盾カードは実データで出す（デモ用に「言葉」と食い違う画像を用意）。
- [ ] 音声は聞き取りやすく（ヘッドホン推奨、エコー対策）。字幕を必ず載せる（無音再生でも伝わる）。
- [ ] 画面のトークン・メール・API キー等が**映り込まない**こと（提出は公開）。
- [ ] 60 秒を超えないこと。超えるなら カット1・7 を各 2 秒詰める。
- [ ] 書き出し: 1080p / mp4 / 字幕焼き込み。ProtoPedia 埋め込み（YouTube 限定公開 or 直接アップロード）。

## ナレーション原稿（読み上げ用・約 110 字）

> コードを書く時間は短くなった。今のボトルネックは要件定義です。SANBA は、話すだけで次に聞くべき問いを自律的に選び、見せた画面から言葉と画の矛盾まで検知し、専門エージェントが要件を構造化します。最後は人間が承認して確定。CI/CD も観測性も DORA も回す、"届く"要件定義エージェントです。
