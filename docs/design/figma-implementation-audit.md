## A. 認証ゲート仕様の検証（明示要件）

> 要件: **「ログインしていれば全画面にアクセスでき、ログインしていなければログイン画面に戻る」**

### 結論

- **「ログイン時は全画面アクセス可」… ほぼ成立。** ログイン済みなら `/`（ホーム→準備→会話）・`/admin` に到達できる。
  ただし `/admin` は **認可（管理者）** を API 側 `ADMIN_EMAILS` で判定し、非管理者は 403 ゲート表示になる（これは認証ではなく認可で正しい）。
- **「未ログイン時はログイン画面に戻る」… 未達（要対応）。** どのページも `/login` への**自動リダイレクトをしていない**。
  - `RequireAuth`（`components/RequireAuth.tsx`）は未ログイン時に `router.replace('/login?next=...')` する正しい部品で、
    単体テスト（`RequireAuth.test.tsx`）も通っているが、**どのページからも import されていない（＝死蔵）**。
  - `app/` 配下に `middleware.ts` も無く、ルート保護は実質「無し」。

### ルート別の実態

| ルート | 未ログイン時の挙動 | 期待（要件） | 判定 |
|---|---|---|---|
| `/login` | ログイン画面（公開） | 公開でよい | OK |
| `/`（01 ホーム / 02 準備） | **画面が見える**。会話開始ボタンのみ無効化＋「ログインへ」リンク表示。リダイレクトしない | /login へ戻す | **未達** |
| `/`（03–08 会話フェーズ） | 開始操作（`handleStart`）が `auth.credential` 必須のため**実質到達不可**。ただしゲートは「開始ボタン無効」止まりで、ルートとしては保護されていない | /login へ戻す | 実質ガード/形式は未保護 |
| `/admin` | 自前の access ステートで **「401 ログインが必要です」ゲート＋手動リンク**を表示。自動リダイレクトはしない | /login へ戻す | **部分**（戻る導線はあるが自動遷移なし） |
| `/design` | 開発専用（`NODE_ENV==='production'` で `notFound()`） | — | OK |

### 設計意図との衝突（要判断ポイント）

- `app/page.tsx` のコメントと `app/page.test.tsx` は **ホームを公開ランディングとして意図**している
  （テスト「未ログインでは開始が無効でログイン導線を示す」が、未ログインでもホーム/準備が見えることを前提に固定）。
- よって「**全画面**でリダイレクト」を厳密採用すると、この意図とテストに反する。
  - **方針案1（厳密）**: 全保護ルート（`/`, `/admin`）を `RequireAuth` でラップし未ログインは `/login?next=` へ。ホームのランディング性は失う。
  - **方針案2（折衷）**: ホームは公開ランディングのまま、**会話フェーズと `/admin` のみ** `RequireAuth` で保護し自動リダイレクト。
    死蔵していた `RequireAuth` を本来の用途で結線する形。docs/conversation-experience.md §3「保護ルート」の記述とも整合。

→ いずれにせよ現状は「未ログイン→/login自動リダイレクト」が**1ルートも実装されていない**ため要対応。採用方針は人間判断（CLAUDE.md 原則1）。
