● マルチモーダル I/O — 提供 ⇄ 提示

見せて、聴いて、描いて返す。

要件は言葉だけに宿らない。SANBA はユーザーから画像・動画・画面共有を受け取って要件を抽出し、 逆に確定要件から完成イメージのビジュアルを生成して提示する。「これで合ってますか？」と問い返すことで、 言葉にならない差分まで引き出す ── ビジュアルを道具にした産婆術。

入力 画像 / 動画 / 画面共有 / カメラ 核 言葉 × ビジュアルの矛盾検知 出力 完成イメージ生成（仮説） 還流 「違う」→ 要件更新
01 — The Multimodal Loop

受け取る → 検知する → 描いて返す → 直す

音声・ビジュアルの入力が要件を太らせ、要件が完成イメージを生み、そのイメージへのフィードバックがまた要件を磨く。一方通行でなく循環させる。

🎤🖼🎥入力（提供）音声＋画像/動画/画面共有。ビジュアルから要件を抽出
→
⚡矛盾・抜け検知言葉 × ビジュアルの食い違いを検出（核）
→
📋確定要件出所・確信度つきで構造化
→
✨完成イメージ生成（提示）「こんな画面になりそうです」を可視化
↩︎ フィードバックの還流：生成イメージの「ここが違う」を新しい発話として取り込み、検知 → 要件更新へ戻す（産婆術のループ）。
02 — INPUT ｜ ユーザーからの情報提供

どんな形でも受け取る

準備時にも対話中にも、画像・動画・画面共有・カメラで素材を渡せる。SANBA はそこから要件を抽出し、言葉との矛盾を検知する（ADR-0004）。

I1

形を問わない

画像・動画・ライブ画面共有・カメラ撮影（ホワイトボード/スケッチ）に対応。

I2

ビジュアル→要件

画面要素・画面内テキストを要件化。出所は「どのアセットの何秒か」まで保持。

I3

言葉×画の矛盾

「検索したいと言ったが画面に検索バーが無い」を検知し核に接続。

I4

動画は時間軸で

フレーム/モーメント単位に検知マーカー。タイムラインで根拠を指し示す。

03 — OUTPUT ｜ SANBA からの完成イメージ提示

確定要件から、画面を描いて返す

「現時点の要件だと、こんな画面になりそうです」。生成したビジュアルを問いの道具として提示し、言葉にならない差分を引き出す。要件→画面イメージ→コードを地続きにする（ADR-0011）。

O1

生成物は「仮説」

「AI生成・推測」を必ず明示。事実や確定として提示しない。

O2

要素をたどれる

各UI要素が「どの要件から来たか」を明示。未確定はゴースト表示。

O3

問い返しの道具

「ここが違う」を引き出し、差分を発話＝要件として還流させる。

O4

確定は人間

採用判断・Figma/実装への受け渡しは人手のレビュー後（CLAUDE.md）。

04 — 実装メモ（設計の接続）

既存アーキテクチャへの載せ方

入力経路

画面共有/カメラは既存の LiveKit 映像トラック（ADR-0004）。画像/動画アップロードは API → Cloud Storage → Gemini マルチモーダル解析。出所メタを付与し確定要件へ。

生成経路

確定要件（Firestore）→ プロンプト整形 → 画像/UI生成 → プレビュー。生成は非同期で会話を止めない（二層構造を踏襲, ADR-0002）。

観測 & 誠実さ

生成の所要時間/コストは Langfuse で観測。生成物は「仮説」ラベル必須・確定は人間（薄い実装の禁止／成果物責任は人間, CLAUDE.md）。要 ADR 化。

※「完成イメージ生成」はプロダクト/アーキの拡張のため、採用時は docs/adr/NNNN-generative-preview.md として設計判断を記録する想定。
