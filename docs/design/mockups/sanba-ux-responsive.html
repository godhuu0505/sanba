● レスポンシブ — スマホ / タブレット版

片手で、声で、解像度を上げる。

移動中・会議前のハンズフリー利用を想定し、SANBA を音声ファースト＋ボトムシートで再設計。 デスクトップの3分割（声 / 検知 / 要件）を、小さな画面では「声を全面に出し、検知・要件は必要なときだけせり上がる」体験へ畳む。

📱 Phone 375–430px 📐 Tablet 768–1024px パターン Bottom Sheet + Tab Bar 親指で届く Thumb-zone 設計
01 — Responsive Principles

小さな画面の設計指針

「全部見せる」をやめ、声を主役に。検知（核）はトーストとボトムシートで割り込み、要件はタブで引き出す。

M1

声を全面に

常時はオーブ＋字幕がほぼ全画面。テキストUIは畳んで、歩きながら・準備しながら使える。

M2

検知はせり上がる

矛盾・抜けを検知した瞬間だけボトムシートが peek→展開。会話を止めず最小の割り込みで伝える。

M3

親指で完結

主要操作（マイク・タブ・確定）は下部 thumb-zone に。44px 以上のタップ領域、片手前提。

M4

1画面1ジョブ

準備→対話→要件→サマリを縦スクロールの単一フローに。同時表示は最小、文脈は維持。

02 — Layout Adaptation

3分割をどう畳むか

同じ情報構造を、画面幅に応じて「並べる / 切り替える / 重ねる」で出し分ける。

🖥 Desktop ≥1200

声・検知・要件を横3分割で常時表示。情報量を最大化。

📐 Tablet 768–1024

声を左、検知⇄要件をセグメントで切替える右ペインの2分割。

📱 Phone ≤430

声を全面。検知=ボトムシート、要件=タブで重ねる。

03 — Phone

スマホ版 主要画面

横スクロールで5画面。核となる「ライブ対話」は ①音声 → ②検知シート展開 → ③要件タブ の3状態で示す。

1ボトムシート＝核の器

矛盾・抜けを peek（1行）→ 展開（全幅）で。会話を止めず最小割り込み。デスクトップ中央列の役割を担う。

↳ ADR-0008 #2 / M2
2下タブ + 中央マイク

対話/検知/要件を親指で切替。中央のマイクは録音トグル。検知バッジで未読件数を可視化。

↳ M3 thumb-zone
3確定操作は sticky

「始める」「確定」など主要CTAは画面下に固定し、長い縦スクロールでも常に届く。

↳ M3 / M4
